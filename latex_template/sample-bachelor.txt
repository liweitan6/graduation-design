                           摘   要
   深度学习框架（如 PyTorch、TensorFlow）已成为人工智能应用中的基础软件，其底
层算子实现的可靠性会直接影响上层模型训练与推理结果。模糊测试（Fuzzing）可用于
发现这类框架中的底层缺陷，但在实际测试中仍存在用例冗余、崩溃日志难以分析以及
生成过程缺少覆盖反馈等问题：大量相似或低收益用例会增加执行开销，非结构化长堆
栈需要人工结合算子语义和运行环境判断根因，局部参数空间内的重复变异也会降低测
试推进效率。
   本文设计并实现了一套结合语义检索与大语言模型（LLM）的模糊测试辅助分析
平台。系统采用前后端分离架构，前端使用 Vue 3 实现交互界面，算法服务由 Python
FastAPI 承载，业务数据管理由 Java Spring Boot 提供，并使用 MySQL 与 Milvus 分别管
理结构化元数据和高维语义向量。平台将测试用例、覆盖信息、崩溃日志和语义检索结
果组织到同一分析流程中，用于支持用例筛选、故障定位以及报告和补盲脚本生成。
   为减少低收益用例的执行开销，本文构建了覆盖效能指数（Coverage Efficiency In-
dex, CEI）模型。CEI 综合考虑单个测试用例的分支覆盖贡献和加权计算成本，通过动态
百分位阈值筛选低收益用例。在 2000 例模拟实验中，该方法在覆盖率损失不足 4% 的
情况下将执行耗时缩减 60%；在 1000 例真实物理实验中，Gcov/LCOV 探针采集的物理
命中行数仅下降 0.005%，总执行耗时缩减 34.5%。
   针对崩溃根因定位，系统实现了双空间 Daikon 边界定界引擎，即结合语义相似用
例召回和参数约束推断来寻找可能触发崩溃的边界条件。该引擎召回与目标崩溃用例拓
扑相同的健康组与崩溃组样本，将嵌套张量参数展平为一维数值键值对，并在多模板扫
描框架下执行不变量推断。只有当某一数学约束在健康组样本中全部满足、在崩溃组样
本中全部不满足时，系统才将其标记为可能导致崩溃的约束边界。实验结果显示，该引
擎在五条已知约束上的召回率为 100%，并能在真实数据中发现跨算子的隐式整除约束
等非平凡边界条件，为崩溃原因分析提供了比堆栈日志更具体的参数依据。
              北京航空航天大学毕业设计 (论文)                     第 II 页


  平台进一步引入检索增强生成（Retrieval-Augmented Generation, RAG）方法，将相
似用例上下文和 Daikon 推断出的约束边界写入结构化四板块提示词模板，由 LLM 生成
包含相关用例、可疑约束和复现线索的诊断报告，同时生成靶向补盲测试脚本。与直接
使用 LLM 输出相比，RAG 辅助诊断的准确性评分提高 50%，幻觉程度降低 61%。
  此外，系统实现了基于 multiprocessing.spawn 的进程级隔离执行引擎，通过捕获子
进程退出状态码拦截底层 C++ 段错误与断言失败，避免单个异常用例导致主服务中断。


关键词：深度学习框架，模糊测试，语义检索，检索增强生成，覆盖效能指数

              Intelligent Management and Analysis of Test Cases
                       for Deep Learning System Fuzzing

                                                                            Author: Guo Junjie
                                                                             Tutor: Wu Ji

                                            Abstract
     Deep learning frameworks such as PyTorch and TensorFlow have become foundational
software infrastructure for artificial intelligence applications. The reliability of their low-level
operator implementations directly affects the correctness of model training and inference. Al-
though fuzz testing can expose defects in such frameworks, its practical use is still limited
by redundant test cases, hard-to-analyze crash logs, and insufficient coverage feedback dur-
ing generation. Similar or low-benefit cases increase execution cost, long unstructured stack
traces require manual reasoning about operator semantics and runtime conditions, and repeated
mutations in a local parameter space may slow down the testing process.
     This thesis designs and implements a fuzz testing auxiliary analysis platform that combines
semantic retrieval with large language models (LLMs). The system follows a decoupled front-
end/back-end architecture: Vue 3 is used for the interactive front end, Python FastAPI provides
the algorithmic service, Java Spring Boot manages business data, and MySQL with Milvus
stores structured metadata and high-dimensional semantic vectors. The platform organizes test
cases, coverage information, crash logs, and semantic retrieval results in one analysis workflow,
supporting test case filtering, fault localization, and the generation of diagnostic reports and
gap-filling scripts.
     To reduce the execution cost of low-benefit cases, this thesis builds a Coverage Efficiency
Index (CEI) model. CEI considers both the branch coverage contribution and the weighted
computational cost of each test case, and filters low-benefit cases using a dynamic percentile
threshold. In a simulated experiment with 2,000 cases, the method reduced execution time by
60% with less than 4% coverage loss. In a physical experiment with 1,000 cases instrumented
by Gcov/LCOV probes, the number of physically hit lines decreased by only 0.005%, while the
total execution time was reduced by 34.5%.
                           北京航空航天大学毕业设计 (论文)                                              第 IV 页


     For crash root-cause localization, the system implements a dual-space Daikon bound-
ary delineation engine, which combines semantic retrieval of similar cases with parameter-
constraint inference to identify possible crash-triggering boundaries. The engine retrieves healthy
and crashed samples that share the same operator topology as the target crash case, flattens
nested tensor parameters into one-dimensional numerical key-value pairs, and performs invari-
ant inference under a multi-template scanning framework. A mathematical constraint is reported
as a possible crash boundary only when it is satisfied by all healthy samples and violated by all
crashed samples. The engine reaches a 100% recall rate on five known constraints and also
discovers non-trivial boundary conditions in real data, including implicit divisibility constraints
across operators. These results provide more concrete parameter evidence for crash analysis
than stack traces alone.
     Retrieval-Augmented Generation (RAG) is then used to connect the retrieved cases and
inferred constraints with LLM-based analysis. The retrieved similar-case contexts and Daikon-
inferred constraint boundaries are inserted into a structured four-block prompt template, so that
the LLM can generate diagnostic reports containing related cases, suspicious constraints, and
reproduction clues, as well as targeted gap-filling test scripts. Compared with direct LLM out-
put, RAG-assisted diagnosis improves the accuracy score by 50% and reduces the hallucination
level by 61%.
     The system also implements a process-level isolated execution engine based on multipro-
cessing.spawn. By capturing child process exit codes, this module intercepts low-level C++
segmentation faults and assertion failures, so that a single abnormal test case does not interrupt
the main service.


Key words: Deep Learning Framework, Fuzz Testing, Semantic Retrieval, Retrieval-Augmented
Generation, Coverage Efficiency Index

                       1   绪论

1.1     课题背景

1.1.1    深度学习框架的工程地位与安全隐患

      深度学习（Deep Learning, DL）框架已经成为人工智能应用开发中的重要软件层。
以 PyTorch 和 TensorFlow 为代表的主流框架，向下封装异构计算硬件、内存分配和算子
调度细节，向上提供数学算子（Operator）库、自动微分接口和模型构建工具。这种封
装方式使开发者能够在较高抽象层次上构建和训练神经网络模型，但也使框架本身成为
由 Python 前端、C++ 运行时、算子内核、自动微分模块和硬件后端共同组成的复杂系
统。当系统规模扩大后，底层算子实现、数据类型转换、边界维度处理和内存管理中的
缺陷，往往难以通过少量人工样例或常规功能测试暴露 [1]。
        与传统应用软件中较容易观察到的异常中断不同，DL 框架中的部分缺陷具有较强
隐蔽性。例如，在特定张量维度组合、广播规则或混合精度计算条件下，底层算子可能
在正向传播或梯度计算过程中产生细小的数值偏差。这类偏差不一定触发运行时错误，
也不一定在单次推理或训练日志中表现为明显异常。上层神经网络如果在包含此类缺陷
的框架上完成训练，可能会将偏差吸收到权重更新过程中，从而影响最终模型行为 [2]。
在自动驾驶感知、医疗影像分析等对输出稳定性要求较高的应用中，这类问题可能进一
步表现为漏检、误判或结果不一致。除数值错误外，框架底层的内存越界、缓冲区溢出
和断言失败也会带来运行稳定性风险。测试者或攻击者可以通过构造包含异常张量形
状、特殊数据类型和边界参数组合的计算图（Computation Graph）触发非法内存访问，
进而影响云端训练或推理服务的可靠性 [3]。
      因此，DL 框架测试不能只停留在 API 是否能够正常调用的层面。对于同一个算子，
不同输入维度、数据类型、设备后端和参数取值可能对应不同的底层执行路径。尤其是
在算子组合形成复杂计算图后，单个算子内部的边界行为还可能与前后算子的输出形状
或类型约束发生关联。这使得框架测试需要覆盖大量非常规输入，包括极端维度、空张
量、非连续内存布局、异常广播关系和跨算子的参数约束。仅依赖人工经验构造测试样
例，很难持续覆盖这些组合空间。

1.1.2    模糊测试的引入与测试数据管理瓶颈

     针对深度学习框架中由复杂输入空间引发的缺陷，传统基于人工编写固定输入的单
元测试方法存在覆盖不足的问题。框架接口需要同时处理多种数据类型、张量维度、设
备后端以及算子之间的嵌套组合，输入空间很难通过少量手工样例完整枚举。因此，学
术界与工业界逐渐将模糊测试（Fuzzing）引入 DL 框架测试场景 [4]。模糊测试通过自
动化算法生成包含畸形参数、极端边界值或特殊算子组合的输入，并将其持续送入待测
框架执行，以发现普通测试样例不易触发的崩溃、断言失败、数值异常或后端不一致问
题。
    在 DL 框架场景中，模糊测试生成的输入通常不是简单字符串或标量参数，而是能
够直接调用框架 API 的测试脚本。这些脚本中包含随机生成或变异得到的算子拓扑、张
量形状、数据类型、设备配置和属性参数。测试执行后，系统还会同步产生覆盖率数据、
崩溃日志、执行耗时、异常类型、退出状态码和运行环境记录等产物。覆盖率数据记录
测试用例触达的代码分支或物理行；崩溃日志可能混合 Python 调用栈、C++ 后端错误、
底层断言信息和部分硬件相关输出；运行记录则用于说明该用例的执行时间、资源消耗
和是否触发异常。这些数据共同构成后续分析的基础，但它们格式差异较大，来源也分
散在脚本文件、日志文本、覆盖率报告和数据库记录中。
    当测试任务长时间运行或并发执行时，这些测试产物会迅速积累。如果缺少统一的
数据组织方式，后续的用例筛选、崩溃归并和根因分析都需要较多人工参与。具体而言，
测试脚本需要与其覆盖率记录、执行结果和崩溃日志建立对应关系；同一底层缺陷触发
的多条崩溃日志需要被归并；覆盖收益较低但执行成本较高的用例需要被识别；历史失
效样本中包含的参数边界和算子组合也需要被重新利用。因此，DL 框架模糊测试的难
点并不只在于生成更多输入，还在于如何管理和分析生成之后的大量测试产物。
     首先，崩溃日志的检索和归类并不适合完全依赖关键字匹配。同一底层缺陷可能因
输入脚本、变量名、张量形状或调用路径不同而产生不同表层日志；相反，日志中相似的
错误词也可能对应不同算子、不同参数组合或不同后端问题。传统文本检索工具主要根
据字面特征匹配，难以同时利用算子拓扑、参数结构和历史相似用例等上下文信息。此
外，DL 框架的底层计算引擎封装了大量调度细节，异常堆栈往往停留在运行时或 C++
内核层，难以直接映射到应用层触发该异常的具体算子组合。这会使开发者需要在测试
脚本、日志文件和框架源码之间反复切换，从而拉长根因定位和缺陷复现的周期 [5]。
        其次，测试用例的多样性需要结合计算图语义来观察。模糊测试的效果依赖于生成
                  北京航空航天大学毕业设计 (论文)                  第   7   页


算法对输入空间的探索广度，但变异引擎在实际运行中容易反复生成结构相近的算子组
合。例如，不同脚本在文本上可能存在较大差异，但其计算图拓扑、张量维度变化和参
数约束实际上高度相似；也可能出现脚本文本相近，但由于数据类型或边界参数不同而
触发不同执行路径的情况。对于包含节点属性、边连接关系和多维张量配置的计算图，
简单统计算子数量、脚本长度或异常次数难以反映测试集的实际分布。传统关系型数据
库可以保存元数据，但不擅长表达高维结构之间的相似关系；基础统计图表也难以展示
测试集在语义空间中的聚集、重复和空缺区域 [6]。因此，测试数据管理需要引入语义
表示和相似检索机制，以支持对历史用例的召回、归类和后续分析。
      最后，单独观察覆盖率也不足以衡量测试用例价值。在模糊测试中，一个用例即使
带来少量新覆盖，也可能消耗较长执行时间或占用较多显存；另一些用例虽然没有触发
崩溃，但可能覆盖了尚未充分探索的算子组合。如果只使用” 代码覆盖率” 或” 是否异
常” 等单一指标，系统难以判断某个用例相对于其执行成本是否值得保留。同时，覆盖
率本身也不能说明一个用例是否与已有用例高度重复，更不能直接指出该用例是否有助
于解释某类崩溃。因此，需要同时考虑覆盖贡献、运行成本和语义差异，对低收益冗余
用例进行筛选，并从历史崩溃样本中提取可复用的约束信息和补盲方向 [7]。这也是本
文引入语义检索、覆盖效能指数（Coverage Efficiency Index, CEI）、双空间 Daikon 约束
分析和检索增强生成（Retrieval-Augmented Generation, RAG）辅助诊断的重要原因。

1.2     国内外研究现状

1.2.1   深度学习框架模糊测试方法研究

      面向 DL 框架的模糊测试研究首先关注如何生成更容易触发异常的输入。Pei 等人
提出的 DeepXplore[8] 引入了神经元覆盖率的概念，通过差分测试策略自动生成能够触
发系统内部逻辑差异的测试输入，为后续深度学习系统测试提供了参考思路。Xie 等人
[9] 提出的 DeepHunter 进一步将变异策略从像素级扰动扩展到覆盖率引导的多层次变异
机制，使测试生成过程能够根据覆盖反馈调整输入变异方向。这类研究主要面向深度学
习模型本身，关注输入样本如何触发模型行为差异或覆盖更多神经元状态。
      在框架级 API 测试方面，Wei 等人提出的 FreeFuzz[10] 从开源代码中提取 API 调用
模式，并据此构造 PyTorch 和 TensorFlow 测试用例。这类方法的优势在于能够利用真实
代码中的调用习惯，以较低人工成本覆盖大量框架接口。Deng 等人 [11] 进一步探索了
利用大语言模型（Large Language Model, LLM）生成框架测试用例的可行性，说明 LLM

能够在一定程度上理解框架 API 签名和参数关系，并生成可执行的测试代码。这些工作
推动了 DL 框架测试输入的自动化生成，尤其是在 API 组合和测试脚本构造方面提供了
有价值的方法。
    不过，从本文关注的问题来看，上述研究的重点仍然集中在生成端。它们主要回答
如何产生更多测试输入、如何提高生成输入的覆盖能力，以及如何利用已有代码或 LLM
降低测试脚本构造成本。当测试工具已经持续产生大量脚本、覆盖率报告和崩溃日志
后，如何组织这些产物、如何归并相似崩溃、如何筛除低收益用例、以及如何把历史失
败样本转化为新的补盲方向，仍需要额外的系统支持。本文的工作因此更侧重生成之后

于历史测试结果的用例筛选、故障辅助定位和候选补盲脚本生成。

1.2.2   测试数据管理与效能评估相关研究

     测试用例管理领域已有较多关于优先级排序和用例约简的研究。Rothermel 等人
[12] 提出了基于覆盖率的贪心排序策略，通过优先执行能够覆盖更多新代码分支的用例
来提升回归测试效率。Harrold 等人 [13] 研究了在维持覆盖率约束下的最小用例集选择
问题。这类方法在传统软件测试中应用较多，其基本假设是覆盖率向量能够较好地表示
测试用例之间的差异，并且测试输入本身可以用相对明确的结构化特征描述。在回归测
试场景中，这一思路能够减少重复执行带来的成本。
    DL 框架模糊测试产物与传统回归测试用例存在明显差别。一方面，测试输入不只
是标量参数或简单对象，而是由算子拓扑、张量形状、数据类型和属性约束共同构成的
高维结构。另一方面，测试执行结果也不只包括通过或失败，还包括覆盖率变化、崩溃
堆栈、执行耗时和后端环境差异。如果仅根据文本相似度或覆盖率向量判断用例价值，
容易忽略计算图结构和参数组合上的语义接近关系。例如，两个用例可能覆盖相近代码
区域，但一个用例执行成本很低，另一个用例需要较长时间和更多显存；也可能出现两
个用例覆盖率差异不大，但其中一个触发了更接近历史崩溃样本的参数边界。因此，用
例约简和效能评估需要引入更适合高维测试数据的表示方式，并同时考虑覆盖贡献与计
算成本。

供了可行工具 [14]。以 Milvus 为代表的向量检索引擎支持对稠密浮点向量执行近似最
近邻（Approximate Nearest Neighbor, ANN）查询，能够在较短响应时间内完成大规模向

量相似度匹配。将语义编码和向量检索用于测试数据管理，可以弥补关键字检索在处理
崩溃日志和计算图脚本时的不足。通过向量化表示，系统可以根据语义相似性召回历史
脚本和崩溃样本，而不仅仅依赖日志中的固定词项或脚本中的字符串片段。
     不过，单独使用向量检索并不能完成覆盖率分析、用例成本评估或约束边界判断。
向量检索更适合回答” 哪些历史样本与当前样本相似”，而不能直接判断该样本是否值得

同时结合相似用例召回和 Daikon 约束分析，为崩溃原因分析提供参数层面的辅助依据。

1.2.3    大语言模型辅助软件测试研究现状

     LLM 在软件工程领域的应用正受到广泛关注 [15]。在代码生成方面，Codex[16] 与
CodeLlama[17] 等模型展示了根据自然语言描述生成程序代码的能力。在缺陷修复方面，

可以参与测试脚本生成、错误解释和候选修复建议等环节。在 DL 框架测试场景中，LLM
尤其适合处理 API 调用模式、参数组合和测试脚本模板等与代码语义相关的问题。
    但是，通用 LLM 在处理 DL 框架故障诊断时仍存在限制。框架崩溃日志通常包含
跨语言调用栈、底层运行时错误和版本相关行为。如果模型无法获得当前项目中的相似
用例、参数配置、覆盖信息和历史执行结果，就容易根据通用经验给出并不适用的解释。

1.3   研究目标与主要工作

      本课题围绕 DL 框架模糊测试产物的管理、分析和再利用展开。系统面向测试过程
中产生的脚本、覆盖率数据、崩溃日志和运行记录，设计一套结合结构化存储、语义检
索、效能评估和 LLM 辅助诊断的分析平台。研究目标不是替代现有模糊测试生成器，而
是在测试输入产生之后，对测试产物进行组织、检索、筛选和诊断，从而提高历史测试

2.1     深度学习框架与模糊测试技术

2.1.1    深度学习框架的体系结构

      当前主流 DL 框架普遍采用分层架构设计 [21]。以 PyTorch 为例，其体系结构自顶
向下可划分为三个层次。最上层是面向用户的 Python 前端接口，提供张量操作、自动微
分以及神经网络模块的高级抽象。中间层负责组织用户定义的运算关系，在执行、追踪
                                             。为便于后续分析，
或优化过程中可形成计算图或中间表示（Intermediate Representation）
本文将算子之间的数据依赖抽象为有向无环图（Directed Acyclic Graph, DAG）结构，其
中节点对应算子调用，有向边描述数据的流向依赖关系。最底层是 C++ 实现的 ATen 张
量计算库与 CUDA 核函数调度引擎，直接处理 GPU 显存分配、并行线程调度与底层数
值运算。
      这种三层结构使框架接口保持较高的易用性，也保证了底层执行效率。但从测试和
故障分析角度看，分层封装会拉开用户脚本与底层算子实现之间的距离。当 ATen 库或
CUDA 核函数中的某个算子在边界输入下发生异常时，上层 Python 堆栈通常只能反映
API 调用路径，未必能直接指出真实触发缺陷的底层实现位置。如果异常还涉及计算图
中前后算子的形状传递、数据类型转换或设备切换，仅依靠日志文本进行人工判断会更
加困难。本文后续设计的故障辅助定位模块和语义检索流程，正是围绕这种跨层封装带
来的分析困难展开。

2.1.2    模糊测试技术原理与分类

      Fuzzing 通过自动化方式持续生成测试输入，将其送入待测程序执行，并观察程序
在运行过程中是否出现异常信号 [22]。依据输入的构造策略，Fuzzing 通常可分为两类：
基于变异（Mutation-based）的方法从已有种子输入出发，通过随机翻转、插入或删除字
节来派生新的测试用例 [23]；基于生成（Generation-based）的方法则依据输入格式的语
法规范或语义模型从零构建符合特定约束的测试输入 [24]。
        在 DL 框架测试场景中，Fuzzing 输入不再只是文件字节流或简单参数，而是能够
直接调用框架 API 的测试脚本。一个用例通常包含算子类型、张量形状、数据类型、设
                 北京航空航天大学毕业设计 (论文)                   第   13   页


备配置以及算子之间的拓扑连接关系。因此，测试系统除了要保存脚本本身，还需要记
录执行结果、覆盖率变化、崩溃日志和运行环境等信息。DL 框架中的” 崩溃” 也不只包
括段错误与未捕获异常。在差分测试场景中，不同后端或不同实现之间出现数值结果不
一致，也可能反映底层算子实现存在问题 [25]。这类异常未必触发操作系统信号，需要
结合执行结果和额外预言机制进行识别。本文系统面向的正是这类结构复杂、来源分散
的 Fuzzing 产物，因此需要在数据摄入阶段对脚本、日志和覆盖率信息进行统一归档与
特征提取。

2.2     文本语义表征与向量检索技术

2.2.1   基于孪生网络的预训练语义编码模型

      传统词袋模型或 TF-IDF 方法主要根据词项出现频率构造稀疏向量，能够表达文本
中出现了哪些关键词，但难以处理词序、上下文语义以及同义表达等问题 [26]。对于 DL
框架崩溃日志而言，同一类底层异常可能因为变量名、张量形状或调用路径不同而呈现
为不同文本；相反，包含相似错误词的日志也可能对应不同算子或不同后端问题。因此，
本文系统需要一种能够表示句子级语义接近关系的编码方法，而不是只依赖字面匹配。
      Reimers 等人提出的 Sentence-BERT[27] 在预训练语言模型的基础上引入孪生网络
（Siamese Network）结构，使句子级嵌入向量之间的距离具有可比较性。具体而言，两
条输入文本分别经过共享参数的编码器后，各自产出一个固定维度的稠密浮点向量，余
弦相似度即可直接反映两段文本的语义接近程度。这一特性适合用于本文系统中的两类
表征：一类是崩溃日志和报错堆栈文本，用于召回历史相似缺陷；另一类是测试脚本中
的结构信息，包括算子拓扑、参数配置和结构摘要等内容，用于比较不同测试用例在计
算图语义上的接近程度。
      本课题选用 all-MiniLM-L6-v2[28] 作为具体的编码器实例。该模型基于 6 层 Trans-
former 架构蒸馏而来，输出维度为 384 维，在保持一定语义区分度的同时具有较低推理
延迟。在本系统中，该模型被部署于数据摄入流水线的向量化环节。每条测试用例入库
前，系统会将日志文本和脚本结构信息编码为稠密语义向量，并将向量标识与结构化元
数据关联保存。这些向量为后续相似用例检索、崩溃归类和测试集多样性分析提供数值
基础。

2.2.2    高维向量近似最近邻检索

      当向量集合规模持续增长时，逐一计算查询向量与库中所有向量之间的距离会带来
较高延迟。近似最近邻（Approximate Nearest Neighbor, ANN）检索通过牺牲少量精度来
换取检索速度提升 [29]。在本文系统中，语义检索需要在历史测试脚本和崩溃日志向量
中快速召回相似样本，因此需要专门的向量检索引擎支撑在线查询。
      Milvus[30] 是一款面向海量向量数据的开源检索引擎，支持多种索引类型与距离度
量方式。本课题选用 IVF_FLAT（Inverted File Index with Flat quantization）作为主索引
结构。该索引首先通过聚类算法将向量空间划分为若干 Voronoi 单元，查询时仅在距离
查询点最近的若干单元内执行精确扫描，从而将搜索范围从全量集合缩减至局部子集
[31]。这种索引方式适合本文系统中” 先召回候选相似用例，再结合结构化字段补全信
息” 的查询流程。

2.3.1    主成分分析原理

2.3.2    Daikon 动态不变量检测框架

     Daikon[33] 由 Ernst 等人提出，用于从程序多次执行的运行时轨迹中推断可能成立
                        。在经典用法中，Daikon 会在被观测程序的关键执行点
的程序不变量（Program Invariant）
采集变量实际取值，然后将这些取值序列与预定义的不变量模板库（如 x > 0、x = a·y+b、
x ∈ {v1 , v2 , . . .} 等）进行匹配。若某一模板在全部已观测轨迹上均成立，则将其报告为一

物中” 成功执行” 样本与” 崩溃触发” 样本两组对照数据。对于同一类算子或相似计算图
结构，系统将张量形状、算子属性和关键参数整理为可比较的数值键值对，再检查哪些
约束在健康样本中普遍成立，却在崩溃样本中被破坏。这些约束不能直接等同于缺陷根
因，但可以作为分析崩溃触发条件的候选边界。第四章将进一步说明本文系统如何结合
语义相似用例召回与 Daikon 式约束推断，构建双空间边界定界算法。

2.4   检索增强生成技术

      RAG 架构 [34] 由检索器（Retriever）与生成器（Generator）两个核心组件构成。当
用户发起查询时，检索器首先从外部知识库中召回与查询语义相关的文档片段，随后将
这些片段作为上下文拼接至 LLM 的输入提示词中，引导生成器在给定上下文范围内产
出回答。在本文系统中，外部知识库并不是通用百科文本，而是历史测试脚本、崩溃日
志、相似用例元数据以及 Daikon 推断得到的候选约束。
      这一架构相较于直接调用 LLM 具有两方面优势。其一，通过外部知识注入减轻模
型对参数记忆的依赖，降低 Hallucination 的发生概率。其二，知识库可动态更新而无需
重新训练模型权重，使系统能够适应持续增长的领域数据 [35]。对于 DL 框架故障诊断
而言，RAG 的作用主要在于把当前崩溃用例与历史相似样本、候选参数边界和覆盖信
息放入同一上下文中，使 LLM 的输出不只依赖通用编程经验。
      本课题在两个场景中集成 RAG 机制：一是故障辅助诊断——以当前崩溃用例的语
义向量召回历史相似案例，并将相关用例、可疑约束和运行信息写入提示词，为 LLM
生成诊断报告提供项目内上下文；二是算子缺口补盲——将识别出的未覆盖或覆盖不足
的算子组合信息注入提示词，由 LLM 生成候选测试脚本。在这两个场景中，RAG 并不
替代确定性分析算法，而是把语义检索、约束分析和覆盖信息组织为更容易阅读的诊断
文本或脚本草案。其具体集成方式将在第四章展开。

2.5   代码覆盖率采集技术

      代码覆盖率用于描述测试用例触达代码区域的情况，是后续计算用例覆盖贡献的重
要数据来源 [36]。GNU 工具链中的 Gcov[37] 是面向 C/C++ 程序的覆盖率采集工具。其
工作原理分为三个阶段：在编译阶段，通过向 GCC 传递–coverage 选项，编译器在源代
码的每个基本块入口处插桩（Instrumentation）计数器指令，同时生成记录源码结构信
息的.gcno 文件；在执行阶段，程序运行时各计数器累加实际命中次数，进程正常退出
后将计数结果落盘为.gcda 文件；在分析阶段，Gcov 读取.gcno 与.gcda 文件，计算并输
出每个源文件的行级与分支级覆盖率。
      LCOV[38] 是对 Gcov 输出的上层封装工具。它支持对多次运行产生的覆盖率数据
执行合并（Merge）与差集运算，并可生成 HTML 格式的可视化覆盖率报告。在本课题
中，系统需要在插桩编译的 PyTorch 底层 C++ 引擎上逐条执行测试用例并采集对应的
覆盖率增量数据。LCOV 的合并与差集能力使系统能够追踪单条用例对全局覆盖率的边
             北京航空航天大学毕业设计 (论文)          第   17   页


际贡献。该数据不是单独用于评价测试充分性，而是与算子权重、算子出现次数等信息
共同输入 CEI 模型，用于判断某条用例相对于其计算成本是否具有保留价值。

为什么崩溃日志难以直接对应到底层触发位置，Fuzzing 技术则提供了持续生成测试脚

评估模块。

3.1     系统需求分析

3.1.1   功能性需求

      根据第一章对 DL 框架 Fuzzing 产物管理问题的分析，本系统需要围绕测试数据接
入、语义检索、效能评估和辅助诊断建立完整的数据处理流程。系统面向的输入主要包
括 Fuzzing 工具生成的测试脚本、执行日志、覆盖贡献数据和运行状态记录，输出则包
括结构化元数据、相似用例检索结果、多样性可视化结果、CEI 评分、候选边界条件以
及诊断报告。具体功能性需求包括以下六项。
      （1）自动化日志解析与特征提取。Fuzzing 工具持续运行后会产生大量原始崩溃日
志与测试脚本，其中既包含 Python 调用栈、C++ 后端错误和退出状态码，也包含算子
名称、张量维度、设备配置和执行耗时等可结构化字段。系统需要在数据摄入阶段解析
这些文件，提取错误类型、触发位置、张量维度配置和运行状态等元数据，并将日志文
本和脚本结构信息编码为稠密语义向量。解析后的结构化字段用于列表筛选、统计和排
序，语义向量则供后续相似用例检索、崩溃归类和多样性分析使用。
      （2）语义检索与故障辅助定位。用户在分析崩溃时，通常无法仅凭关键字确定某条
日志是否与历史缺陷相同。因此，系统需要支持用户输入自然语言描述、错误片段或崩
溃特征，在向量空间中召回语义接近的历史用例。检索结果应同时返回脚本摘要、错误
类型、执行状态、相似度分数和相关元数据，便于用户比较当前崩溃与历史样本之间的
关系。在此基础上，系统还需要结合计算图拓扑关系和错误类型，给出可能相关的算子
节点或参数位置，作为故障辅助定位的参考，而不是替代开发者完成确定性的根因判断。
      （3）测试集多样性监测。Fuzzing 过程容易在局部参数空间或少数算子组合附近重
复变异。为了观察测试集是否出现明显聚集，系统需要将测试用例的高维语义特征降维
到二维平面，并在前端以散点图形式展示。用户可以通过点的空间分布、状态颜色和异
常类型筛选，观察当前测试集在语义空间中的聚集区域、稀疏区域和异常样本位置。该
功能的输出不是单一数值指标，而是为后续用例筛选、补盲方向选择和测试集质量分析
提供可视化依据。
      （4）效能评估与低价值用例清退。在长时间 Fuzzing 任务中，部分测试用例虽然会

占用较多执行资源，但对分支覆盖率或边覆盖率的贡献较低。系统需要建立覆盖贡献与
计算成本相结合的效能量化指标，读取覆盖贡献数据、算子权重和算子出现次数等信息，
计算每条用例的 CEI 得分。用户可以查看单条用例的评分结果，也可以依据动态阈值标
记低收益冗余用例。对于被确认清退的记录，系统需要在 MySQL 结构化记录和 Milvus
向量记录之间执行级联处理，降低单库残留数据对后续检索和统计造成干扰的风险。
   （5）故障边界定界。对于同一类算子或相似计算图结构，健康样本与崩溃样本之间
往往存在参数取值、张量形状或算子属性上的差异。系统需要基于成功执行样本和崩溃
触发样本构造对照组，将嵌套参数整理为可比较的数值键值对，并利用 Daikon 式约束
推断方法寻找候选边界条件。这些候选约束可以帮助用户理解某类崩溃可能与哪些参数
范围有关，但不应被直接表述为确定性根因。前端需要以可读形式展示候选约束、关联
样本和相关参数，供开发者进一步确认。
   （6）智能诊断与算子缺口补盲。系统需要集成 RAG 机制，在崩溃诊断场景中将历
史相似用例、候选约束、运行信息和当前崩溃上下文组织为提示词，为 LLM 生成诊断
文本提供项目内数据依据。诊断结果应包含相关用例、可疑参数条件和复现参考等信
息，帮助用户减少在长日志中手工筛选信息的成本。同时，系统需要根据当前测试集中
未覆盖或覆盖不足的算子组合，构造补盲提示词并生成候选补盲测试脚本。这类脚本作
为后续测试输入的候选材料纳入管理，仍需要经过执行验证和人工检查。

3.1.2   非功能性需求

   （1）高并发写入性能。Fuzzing 任务通常以批量或持续方式产生测试用例，在并发
执行场景下，同一时间可能有多条脚本、日志和覆盖记录进入系统。如果摄入接口写入
能力不足，测试结果会堆积在文件目录或消息队列中，影响后续检索和分析的时效性。
因此，系统的数据摄入接口需要支持每秒数十条用例的并发写入，并在结构化元数据落
盘和语义向量写入之间保持稳定的处理顺序。对于写入失败或部分失败的记录，系统需
要提供重试或异常记录机制，降低数据丢失和双库状态不一致的风险。
   （2）大规模数据可视化性能。多样性监测页面需要在浏览器端展示万级规模的二维
散点图。如果直接依赖普通 DOM 节点或未优化的 Canvas 绘制，在缩放、筛选和悬停查
看详情时容易出现明显卡顿。因此，前端需要采用适合大规模点渲染的方案，在主流硬
件配置下维持较为稳定的交互帧率。这一需求并不只是界面体验问题，还关系到研究人
员能否在大量历史用例中快速观察聚集区域、异常样本和测试空缺。

      （3）底层执行隔离安全性。系统在采集覆盖率数据或验证候选测试脚本时，需要实
际执行由 Fuzzing 产生的测试用例。这类脚本可能触发底层 C++ 段错误、断言失败、死
循环或资源占用异常。如果所有用例都在主服务进程内执行，单个异常样本就可能导致
服务中断，并影响其他用户请求和正在进行的数据写入。因此，每条测试用例需要放入
独立子进程中运行，主进程通过退出状态码和日志捕获获取执行结果，并可结合超时控
制减少长时间挂起对服务的影响，从而隔离底层异常对 Web 服务本身的影响。
      （4）异构数据一致性。本系统同时使用 MySQL 保存结构化元数据，使用 Milvus 保
存高维语义向量。一次数据写入通常需要跨越两个不同存储系统，删除或清退操作也需
要同时处理两边记录。由于两类数据库的事务机制和写入接口不同，系统难以简单依赖
单库事务来覆盖完整流程。因此，设计中需要通过全局唯一标识符、写入状态记录、失
败重试和级联删除策略维护两库关联，尽量降低 MySQL 中存在元数据但 Milvus 缺失向
量，或 Milvus 中残留向量但 MySQL 记录已删除的风险。

维护关系型数据、执行权限校验，并对算法后端提供统一代理入口。测试用例元数据、
错误类型、执行状态、CEI 得分和时间范围等查询条件更适合通过关系型数据库完成组

便于维护业务接口的稳定性。因此，Java 业务后端在系统中承担结构化数据管理和请求

3.2.2   异构双库存储架构

     本系统的数据资产包含两种不同形态：一类是结构化元数据字段，如 case_uid、错
误类型、算子名称、张量维度、时间戳、执行状态、覆盖记录和 CEI 得分等；另一类是由
Sentence-BERT 编码得到的 384 维稠密浮点语义向量，包括日志语义向量和脚本结构信
息对应的向量标识。如果使用单一存储引擎管理这两类数据，要么难以高效完成多条件
                北京航空航天大学毕业设计 (论文)            第   22   页


结构化查询，要么难以支持高维向量的相似度检索。因此，系统采用 MySQL 与 Milvus
协同的异构双库存储架构。
        MySQL 负责保存结构化元数据和业务查询所需字段。例如，测试用例的执行状态、
错误类型、覆盖记录、执行耗时、算子配置摘要、CEI 得分和可视化坐标均以表字段或
JSON 扩展字段的形式保存。这些数据需要支持按状态、时间范围、错误类型、分数区
间和算子名称进行筛选，也需要支持分页、排序和事务性更新。关系型数据库的索引机
制和事务能力更适合处理此类操作。

过 Sentence-BERT 编码后，形成固定维度的稠密向量。这些向量无法通过普通 B+ 树索
引表达语义距离，需要借助 IVF_FLAT 等向量索引进行近似最近邻检索。因此，Milvus
主要用于根据查询向量召回相似日志、相似脚本或候选历史用例，而不承担复杂业务条
件查询和事务性数据维护。
        两库之间通过全局唯一标识符 case_uid 建立关联。每条测试用例入库时，系统首
先生成 case_uid，并将其写入 MySQL 主记录。在向 Milvus 写入日志向量和结构相关向
量时，同一 case_uid 也会作为向量记录的标量字段写入。查询时，系统根据请求类型
选择主查询库：如果用户执行结构化筛选，业务后端先从 MySQL 查询满足条件的用例，
再根据需要补充向量标识或检索结果；如果用户执行语义检索，算法后端先在 Milvus
中召回相似向量及其 case_uid，再回到 MySQL 批量查询对应元数据。最终结果通过
case_uid 完成拼接后返回前端。这种设计能够在结构化查询和语义检索之间取得较好
的功能分工，同时通过统一标识降低双库数据不一致带来的影响。

故障辅助定位模块、多样性监测模块、效能评估与清退模块、边界定界模块以及智能诊
断与补盲模块。其中，语义编码、向量检索、PCA 降维、Daikon 边界分析和 RAG 上下

接口或转发至算法后端，再将处理结果统一组织后返回前端。这种划分使数据接入、检
索分析、效能评估和诊断生成之间保持相对清晰的边界，也便于在后续实现中分别调试
各类业务逻辑和算法逻辑。

4.1.2    基于 DFS 拓扑回溯的故障辅助定位模块

    日志解析完成后，系统需要将结构化错误信息映射回测试用例的计算图结构，定
位出可能与崩溃相关的候选算子。本系统设计的 FaultLocator 模块基于逆向邻接矩阵与
DFS 回溯机制实现该映射，其输出用于辅助用户缩小排查范围，而不是直接给出确定性
根因。
        逆向邻接矩阵的构建过程如下。每个测试用例包含一条算子连接链，以有向箭
                 北京航空航天大学毕业设计 (论文)                           第   28   页


头分隔的字符串形式表示，例如 Conv2d->BatchNorm2d->ReLU->Linear。引擎首先
对该字符串进行分词，以-> 为分隔符解析出有序算子列表。设解析后的算子序列为
{o0 , o1 , . . . , on−1 }，引擎为每个算子 oi 构建索引元组 (operator, index)。在正向邻接关系中，
oi 指向 oi+1（0 ≤ i < n − 1）
                        ；在逆向邻接矩阵中，oi+1 反向指向 oi 。逆向矩阵的构建使得
引擎能够从错误触发点出发，沿数据流反向回溯至上游相关算子。
        每个算子节点的出度（Degree）定义为该节点在逆向邻接矩阵中的后继数量。对于
线性连接链，所有节点的出度均为 1；对于包含残差连接或多分支结构的计算图，分支
汇聚节点的出度大于 1。DFS 回溯在以下条件之一满足时终止：（a）当前节点属于错误
相关算子集合；（b）回溯深度超过连接链总长度的二分之一；（c）当前节点出度为零
（即已到达计算图输入端）。
        引擎针对六种错误类别设计了差异化的定位策略，各策略依据不同的算子能力标注
集合筛选候选对象。维度变换算子集合包含 Linear、Conv2d、MaxPool2d、Flatten、Reshape
等；高显存消耗算子集合包含 Conv2d、Conv3d、MultiHeadAttention、LSTM 等；索引操
作算子集合包含 Embedding、LSTM、Concat 等；设备敏感算子集合包含 Conv2d、Linear、
BatchNorm2d 等。
        以 shape_mismatch 错误为例，引擎在连接链中扫描所有属于维度变换集合的算子，
初始置信度设为 0.85。若日志中同时包含具体的张量维度信息且候选算子为 Linear 或
Conv2d，则置信度提升至 0.95。引擎还检查特殊拓扑模式：若 Conv2d 与 Linear 直接相
邻而中间缺少 Flatten 层，则对 Linear 节点的置信度追加 0.05（上限为 1.0），因为这是
DL 框架中常见的维度不匹配场景。
    最终输出的候选算子列表按置信度降序排列，每个条目包含算子类型、在连接链中
的索引位置、置信度数值与自然语言形式的推理理由。该列表将传递给前端在计算图可
视化中高亮标注候选节点，供用户结合日志、源码和复现结果进一步确认。

4.2.2    异步协程削峰与高并发双写一致性保障

     在批量 Fuzzing 场景下，测试执行引擎可能在短时间内产生大量崩溃记录。若每条
记录都同步触发一次 MySQL 写入加两次 Milvus 写入，网络往返延迟将成为系统吞吐
量的瓶颈。为此，本系统基于 Python 的 asyncio 事件循环设计了内存缓冲队列（Buffer
Queue）削峰机制。
    该机制的核心数据结构为一个容量上限为 B 的异步队列（默认 B = 256）。数据摄
入 API 在接收到请求后，立即将待处理记录推入队列并返回确认响应，从而将 API 响应
延迟与实际入库延迟解耦。
        当队列中的记录数量达到批量阈值 T（默认 T = 64）时，消费者协程被唤醒并执行
一次批量落盘操作。此外，系统还注册了一个周期为 ∆t（默认 ∆t = 5 秒）的定时器协
程。即使队列未满，定时器到期时也会将队列中所有积累的记录一次性刷入数据库。这
种” 容量触发 + 时间触发” 的双重机制兼顾高并发场景下的吞吐量与低负载场景下的数
据及时性，同时避免记录长时间滞留在内存中。
              生产者协程进入背压（Backpressure）等待状态。
        当队列已满时，                          asyncio 的 Queue.put()
方法在队列容量已达上限时自动挂起当前协程，直到消费者取走部分记录后才恢复执
行。该机制以协程级别的非阻塞挂起替代了传统的线程阻塞，避免了线程池资源的浪
费。
        批量落盘操作内部的执行顺序遵守” 先 Milvus 后 MySQL” 原则。原因在于 Milvus
的插入操作返回自动生成的向量主键，该主键需嵌入 MySQL 记录的 parameters 字段中。
若 MySQL 先行写入则缺少向量主键信息，可能导致跨库关联不完整。两步操作之间不
引入分布式事务机制，而是通过以下补偿策略维护最终一致性：若 Milvus 写入成功但
MySQL 写入失败，系统将失败记录重新推入队列尾部并记录告警日志；若连续三次重
试仍失败，该批次记录被转移至死信队列等待人工介入。该机制用于降低双库数据不一
致风险，不等同于严格意义上的分布式事务保证。

4.2.3    基于 CEI 阈值的跨库级联清理机制

      随着 Fuzzing 持续运行，数据库中将积累大量低价值的冗余测试用例。如第三章所
述，本系统通过 CEI 指标量化每个用例的覆盖效能贡献。当用户触发清退操作时，系统
需要在 MySQL 与 Milvus 之间执行基于向量主键的级联删除流程。
        动态阈值的计算采用中位数策略。系统首先从 MySQL 中查询所有用例的 CEI 值并
计算其中位数 c̃，低于 α · c̃（默认 α = 0.5）的用例被标记为清退候选。中位数相较于均
值对极端离群值具有更强的鲁棒性，避免了少数高 CEI 用例拉高阈值导致误删的风险。
      级联删除流程分为三步。第一步，从 MySQL 中批量查询所有满足清退条件的记录，
提取其 vector_id、structure_vector_id 与 log_vector_id。第二步，调用 Milvus 的 delete
接口按主键批量删除两个集合中的对应向量。第三步，在 MySQL 中执行 DELETE 语
句移除对应记录。
      为减少脏读，删除操作期间系统对涉及的 case_uid 集合加入内存级互斥标记。检索
接口在返回结果前过滤掉所有带有删除标记的用例，降低用户在清退过程中看到即将
被移除数据的风险。Milvus 侧的删除操作需在执行后调用 collection.flush() 强制持久化，
否则已删除的向量在 Segment 合并之前仍可能被检索命中。由于 MySQL 与 Milvus 不共
享事务管理器，该机制属于级联清退与最终一致性维护，而不是严格意义上的分布式事
务保证。

4.3.2    候选约束扫描与边界条件推断

     本小节阐述引擎的核心判定逻辑。引擎在公共键集合 K 上系统性地扫描三层模板
池，逐层递增变量数量与公式复杂度。仅当某条候选约束同时满足” 成功全票通过” 与”
失败全票否决” 的双重验证判据时，该约束才被加入候选边界条件集合。这些候选边界
条件反映当前对照样本中的参数差异，不能直接等同于确定性根因。
        第一层：单变量约束扫描。对 K 中的每个键 x，引擎测试以下模板：
               北京航空航天大学毕业设计 (论文)                        第   33   页


  （a）常数阈值模板。取阈值集合 V = {0, 1, −1}，逐一验证 x > v、x ≥ v、x = v 三种
关系是否满足双重判据。
  （b）奇偶性模板。验证 x mod 2 = 0 是否在健康组全部成立且在崩溃组全部违反。该
                   例如 DL 框架中的 GroupNorm、PixelShuffle
模板对应部分算子的通道数或分组参数约束，
等算子对通道数存在整除关系要求。
   第二层：双变量关系约束扫描。对 K 中的任意两个键 x 和 y（x , y），引擎测试以
下模板：
  （a）基础关系模板：x = y、x ≤ y、x < y。
  （b）整除关系模板：x mod y = 0（要求 y , 0）。该模板对应卷积分组操作中的整除
约束，例如 in_channels 能被 groups 整除。
  （c）线性关系模板：x ≤ c · y + d 以及 x ≥ c · y + d，其中系数 c ∈ {2, 3, 4}，偏移
d ∈ {0, 1, −1}。该模板覆盖了 DL 框架中常见的倍数关系约束，例如转置卷积的输出尺寸
公式中的 2 倍因子。
   第三层：三变量约束扫描。对 K 中的任意三个互不相同的键 x、y、z，引擎测试以
下模板：
  （a）空间坍缩公式：x+2y ≥ z。该模板对应 CNN 中的有效感受野约束——当 input_-
size + 2 * padding < kernel_size 时，卷积输出尺寸可能降为零或负值，触发框架运行
时异常。
  （b）张量维度乘积守恒：x · y = z。该模板对应 Reshape 操作的元素总量守恒约束
——重塑前后张量的元素总数必须相等。
   上述三层扫描的双重验证判据以算法1的伪代码形式给出。
   算法1的时间复杂度分析如下。设公共键数量为 n，对照组总样本数为 N = |S| + |F |。
单变量模板的候选谓词数量为 O(n)，双变量模板为 O(n2 )，三变量模板为 O(n3 )。每个候
选谓词需遍历 N 个样本进行验证。因此总时间复杂度为 O(n3 · N)。在实际场景中，展平
后的公共键数量通常不超过 30，对照组总样本数为 20，因此单次定界的计算耗时在毫
秒量级。
   提取出的约束集合 R 经去重后按优先级排序：多变量约束排在单变量约束之前，因
为多变量约束包含更丰富的变量间交互信息，对崩溃条件的解释力更强。排序后的约束
列表即为该崩溃用例的候选边界条件集合，用于为开发者提供后续排查线索。

4.3.3    约束解释与诊断文本生成

      候选边界条件以形式化公式表达，便于程序验证，但对非专家用户缺乏可读性。本
系统将 Daikon 式约束推断结果与 RAG 诊断文本生成相结合，把形式化约束、目标崩溃
用例和相似历史样本组织为 LLM 上下文。
      约束推断阶段即上述 Daikon 定界引擎，输出形如”layers.conv1.kernel_size + 2 * lay-
ers.conv1.padding[0] >= layers.conv1.input_size” 的候选数学约束。
        诊断文本生成阶段基于 RAG 机制调用 LLM。引擎将候选约束集合 R、目标崩溃用
例的结构化元数据、以及召回的相似历史用例上下文组装为结构化提示词模板，注入
LLM。提示词的核心指令要求 LLM 完成三项任务：（1）将候选约束翻译为面向开发者
       （2）结合具体的参数数值解释违反约束的可能含义；
的自然语言解读；                      （3）给出可供人
工检查的参数修正建议或候选安全参数组合。
        LLM 的输出经格式化后与约束推断结果并列展示于前端界面。用户可同时获得可
机器验证的数学公式与可阅读的诊断文本草案。这种设计降低了单独依赖 LLM 可能产
生的幻觉风险：候选约束由确定性算法从真实样本中提取，LLM 主要负责解释和组织
文本，其输出仍需结合源码和复现实验确认。
        在工程实现上，两部分采用串行模式：约束推断先行完成并输出约束集合 R，随后
诊断文本生成阶段将 R 作为上下文的一部分送入 LLM。串行设计使生成内容围绕已有
候选约束展开，减少 LLM 输出偏离实际样本数据的风险。

4.4     测试集多样性度量与 CEI 效能评估模型

4.4.1    基于 PCA 的高维特征降维与空间分布监测

      Fuzzing 过程持续产出的测试用例在高维语义空间中的分布状态，可以反映测试集
在算子组合和参数结构上的多样性。然而 384 维向量空间无法被直接观测，需要降维投
影至二维平面供人类判读。
        本系统在批量更新后或用户请求多样性分析时，对当前样本集合统一执行 PCA 降
维。具体流程为：取当前参与分析的测试用例结构向量矩阵 X ∈ Rb×384 （b 为参与降维
的样本数量），计算协方差矩阵并提取前两个主成分方向，将每个样本投影至二维坐标
(xi , yi )。投影后的坐标经最小-最大归一化映射至 [−1, 1] 区间，存入 MySQL 的 parameters
字段中的 vis_x 与 vis_y 键。

     当样本数不足 2 时，PCA 无法提取有效主成分方向。此时系统退化为随机微扰策
略，为每个样本生成幅度为 0.01 的随机坐标，避免前端散点图中所有点重叠于原点。当
样本数恰好为 2 时，PCA 仅能提取一个主成分，系统将第二维度填充为零后仍执行归一

未被充分测试的算子拓扑组合，可作为后续补盲策略的参考方向。

4.4.3   算子组合缺口的关联分析机制

     除对单个用例的效能评估外，系统还需在全局视角下分析” 测试盲区”——即 DL 框
架支持但当前测试集尚未覆盖的算子组合模式。
    系统维护一张标准算子库（Standard Operator Library）O，包含目标 DL 框架所有公
开注册的算子类型。以 PyTorch 为例，O 涵盖卷积族、线性族、循环族、注意力族、归
一化族、池化族、激活族与结构操作族共八大类约 60 种算子。
    全局邻接矩阵 A ∈ {0, 1}|O|×|O| 记录当前测试集中已出现的算子对组合关系。具体构建
方法为：遍历 MySQL 中所有测试用例的连接链，对于每条连接链 o0 → o1 → · · · → ok−1
中的每对相邻算子 (oi , oi+1 )，将 A[oi ][oi+1 ] 置为 1。遍历完成后，A 中值为 1 的元素对应
已覆盖的算子对，值为 0 的元素对应尚未被任何用例触及的算子对。
    盲区提取通过集合差运算实现。定义理论可行算子对全集 Eall 为标准算子库中所有
在 DL 框架语义下合法的前后衔接组合（排除物理上不可能的对，例如 Softmax 不能直
接接 Conv2d 的通道输入端）。已覆盖算子对集合 Ecovered = {(i, j) | A[i][ j] = 1}。盲区集
合定义为：
                             B = Eall \ Ecovered                              (4.4)

    盲区集合 B 中的每个元素 (oi , o j ) 代表一对尚未被测试的算子衔接。系统将 B 按算
子类别分组后输出为结构化列表，每个条目包含源算子类型、目标算子类型以及该对组
合在框架文档中的参数约束摘要。该列表将作为后续 RAG 补盲模块的输入靶标。

4.5     基于 RAG 的辅助诊断与候选补盲机制

4.5.1    环境感知辅助诊断

      当用户针对某一崩溃用例发起诊断请求时，系统启动环境感知辅助诊断流程。该流
程由四个串行阶段构成。
      第一阶段为特征向量化。系统将用户查询文本经 Sentence-BERT 编码为 384 维向
量。
      第二阶段为双路并行检索。系统在 fuzzing_structure 与 fuzzing_logs 两个 Milvus
集合中分别执行余弦相似度检索，各返回前 3K 个候选结果（K 为最终返回数量）。
        第三阶段为动态权重融合。系统根据查询文本中包含的结构关键词与日志关键词比
例，动态调节两路检索得分的融合权重：若查询同时包含结构关键词与日志关键词，双
路权重各取 0.5；若仅包含结构关键词，结构路权重提升至 0.8；若仅包含日志关键词，
日志路权重提升至 0.8。融合后按得分降序截取前 K 个结果。
        第四阶段为上下文组装与 LLM 推理。系统从 MySQL 中反查召回用例的完整元数
据，将 case_uid、执行状态、边覆盖数、执行耗时、错误信息与算子列表拼装为结构化
上下文文本，连同用户原始查询一并注入 LLM。LLM 基于检索到的真实历史案例生成
诊断文本草案，用于提示相似用例、可疑参数条件和排查方向。该输出仍需要用户结合
源码和复现实验进行检查。

4.5.2    算子缺口靶向补盲与候选用例生成

      盲区集合 B 标识了当前测试集的覆盖缺口。本系统通过构造结构化提示词，引导
LLM 生成候选补盲脚本草案以覆盖这些缺口。

      上述四板块模板的工程意义在于：System 板块限定 LLM 的角色边界，减少其偏离

5.4.1    Gcov/LCOV 探针的编译挂载流程

     Gcov 探针的工作原理基于编译期插桩与运行期计数两阶段协作。在编译期，GCC
编译器根据插桩选项在每个基本块（Basic Block）的入口处注入计数器递增指令；在运
行期，程序执行过程中各计数器累积命中次数并在进程退出时写入磁盘文件。
    为对 PyTorch 底层 C++ 源码启用 Gcov 插桩，需在 WSL 环境中修改编译选项。具
体方式为设置如下环境变量后重新触发 CMake 配置与编译：

export CXXFLAGS=”-fprofile-arcs -ftest-coverage”
export LDFLAGS=”-lgcov --coverage”

     其中-fprofile-arcs 指令使编译器在每个分支跳转点插入弧计数器，-ftest-coverage 指
令使编译器为每个编译单元生成.gcno 文件。链接阶段的-lgcov 参数将 Gcov 运行时库静
态链接至目标二进制。
    编译完成后，每个.cpp 源文件对应产生一个.gcno 文件。该文件记录源码的控制流
图拓扑——包括基本块编号、块间有向边关系与源码行号映射。.gcno 文件在编译时一
次性生成，后续不再变化。
        当携带探针的 PyTorch 库被测试脚本加载并执行后，运行时库在进程正常退出（或
收到 SIGTERM 信号）时，将每个弧计数器的累积值写入与.gcno 同目录的.gcda 文件。

.gcda 文件记录每条边在本次执行中被穿越的次数。多次执行产生的.gcda 数据会累积叠
加。

5.4.2    基于 multiprocessing.spawn 的隔离执行引擎

     测试脚本的执行环境面临两类致命风险：其一，脚本可能触发底层 C++ 算子的段
                     ，导致整个进程地址空间被操作系统回收；其二，PyTorch 内
错误（Segmentation Fault）
部依赖的 OpenMP 线程池在 fork 模式下可能继承父进程的锁状态，造成不可恢复的死
锁。若测试脚本与主 Web 服务共享进程空间，任一风险均将导致后端服务宕机。
     为降低上述风险，本系统采用 multiprocessing 模块的 spawn 启动方式。与默认的
fork 模式不同，spawn 不复制父进程的内存页表，而是启动一个全新的 Python 解释器进
程，从零开始导入目标模块并执行指定函数。该机制确保子进程持有完全独立的地址空
间、线程池与 GPU 上下文。
        启动方式通过调用 multiprocessing.set_start_method(’spawn’, force=True) 全局设
定。force=True 参数覆盖可能已被第三方库设置的启动模式。此后所有通过 multipro-
cessing.Process 创建的子进程均以 spawn 方式启动。
        每个测试脚本的执行流程如下。主进程构造 Process 对象，将执行函数与参数序列
化后传入：

p = multiprocessing.Process(
                         北京航空航天大学毕业设计 (论文)                     第   57   页


    target=_execute_wrapper,
    args=(case_uid, input_shape_str, operators_str)
)
p.start()
p.join(timeout=10)

     p.join(timeout=10) 使主进程阻塞等待子进程完成，最长等待 10 秒。若子进程在超
时时限内未退出，主进程调用 p.terminate() 发送 SIGTERM 信号强制终止子进程，随
后调用 p.join() 等待资源回收完成。当前物理执行脚本已经实现这一超时控制，用于防
止因死循环或底层算子长时间阻塞导致的资源泄漏。
     子进程退出后，主进程通过 p.exitcode 属性获取退出状态码。退出状态码的语义遵
循 POSIX 标准：值为 0 表示正常退出；正整数表示 Python 层未捕获异常导致的退出（如
值 1 为通用错误）；负整数表示子进程被操作系统信号终止，其绝对值为信号编号。
     当底层 C++ 算子因内存越界触发段错误时，Linux 内核向子进程发送 Signal 11
（SIGSEGV）。由于子进程未注册该信号的处理函数，默认行为为立即终止进程并产
生核心转储（若已启用）。此时主进程观察到 p.exitcode == -11。
     主进程可根据退出状态码执行分类处理。当前批量物理执行脚本主要完成隔离执
行、超时回收和探针数据注入；若需要将执行状态回写到数据服务，可采用如下分类逻
辑：

if p.exitcode == 0:
    status = ”SUCCESS”
elif p.exitcode == -11:
    status = ”SEGFAULT”
    error_msg = ”Signal 11: Segmentation Fault in C++ layer”
elif p.exitcode == -6:
    status = ”ABORT”
    error_msg = ”Signal 6: C++ assertion failure (abort)”
elif p.is_alive():
    p.terminate()
    p.join()
    status = ”TIMEOUT”
                    北京航空航天大学毕业设计 (论文)                  第   58   页


else:
   status = ”CRASH”
   error_msg = f”Exit code: {p.exitcode}”

        Signal 11（段错误）与 Signal 6（SIGABRT，断言失败）是 PyTorch 底层算子在非法
参数组合下可能触发的两类信号。通过 p.exitcode 可以将这类异常转换为结构化状态，
再写入 MySQL 的 status 与 error_message 字段，供后续故障辅助定位、候选边界条件
分析和 RAG 诊断使用。
        该隔离机制的核心保障在于：无论子进程以何种方式终止——正常退出、异常退出
或被信号杀死——主进程的地址空间、线程池与网络连接均不受影响。FastAPI 的 ASGI
事件循环持续响应其他 HTTP 请求，系统可用性不因单个测试脚本的 C++ 层崩溃而中
断。
        OpenMP 线程死锁的防护同样依赖 spawn 机制。PyTorch 在首次执行张量运算时初
始化 OpenMP 线程池，线程池内部持有互斥锁保护共享工作队列。在 fork 模式下，子进
程继承父进程的锁副本——若分叉时刻恰好有线程持有该锁，子进程将永久等待一个不
会被释放的锁，形成死锁。spawn 模式通过启动全新解释器彻底规避了该问题：子进程
的 OpenMP 线程池从零初始化，不继承任何锁状态。这是本系统选择 spawn 而非 fork
的根本原因。

6.3.2    约束驱动的反事实样例验证实验

     仅依赖历史成功组与崩溃组之间的统计分离，仍可能将偶然相关的参数关系误判为
故障边界。为进一步验证 Daikon 输出约束的可执行性，本文在边界定界流程中加入约
束驱动的反事实样例生成环节。该环节不再停留于“历史样本是否满足约束”的静态判
断，而是根据约束式主动构造新的测试用例，并将其送入真实算子执行链中验证。
        具体验证流程如下。首先，对 Daikon 输出的每条约束式解析其涉及的数值参数路
径，例如 input_height、kernel_size、padding 或 groups 等。随后，以健康组中的最近
邻成功用例作为基准样例，沿相关参数维度执行小步长变异，分别搜索两类候选样例：
一类满足约束式，记为正样例；另一类违反约束式，记为反样例。候选样例生成后，系
统重新计算约束表达式的布尔值，剔除求值结果与目标类别不一致的样例，保证正反样
例在数学上严格分离。最后，将保留下来的样例交由 PyTorch 算子执行器运行，记录其
实际执行状态与异常摘要。
    对于一条候选约束 C，若其生成的全部正样例均能成功执行，且全部反样例均触发
运行时错误，则认为该约束在当前样例设置下通过执行验证。该判定可表示为：


                ∀x+ ∈ S + , C(x+ ) = true ⇒ Exec(x+ ) = Success,


                ∀x− ∈ S − , C(x− ) = false ⇒ Exec(x− ) = Error.

其中 S + 表示自动生成的正样例集合，S − 表示自动生成的反样例集合。该定义将 Daikon
的静态不变量发现补充为可运行样例检查：约束不仅需要解释历史数据，还需要在新生
成样例上呈现一致的执行结果。

6.3.3    未知边界条件的探索性发现

     已知靶标验证证实了引擎的基础能力，但引擎的核心价值在于发现开发者未曾预见
的隐藏约束。本节报告引擎在真实崩溃数据上给出的一个跨算子候选边界条件案例。
     在对一条包含 Conv1d→GroupNorm 算子链的崩溃用例执行故障边界分析时，引
擎召回了 10 个拓扑相同的健康用例与 8 个崩溃用例。参数展平后，健康组与崩溃组在
Conv1d 的 out_channels 与 GroupNorm 的 num_groups 两个参数上呈现明显分离。
    引擎的双变量整除模板扫描检测到以下约束：

        Conv1d.out_channels % GroupNorm.num_groups == 0

                              ，在全部 8 个崩溃用例中均被
     该约束在全部 10 个健康用例中严格成立（全票通过）
违反（全票否决）。引擎据此将该约束标记为候选边界条件并输出至前端展示面板。
    该约束的物理意义在于：GroupNorm 算子在执行时将输入张量的通道维度均匀切
分为 num_groups 组，每组内部独立计算均值与方差。若 out_channels 不能被 num_-
groups 整除，通道维度无法均匀切分，底层 C++ 实现直接触发断言失败。
        此约束在 PyTorch 官方文档中虽有提及（num_channels 必须能被 num_groups 整
除），但在 Fuzzing 场景下，Conv1d 的输出通道数由上游算子的参数间接决定——当
Conv1d.out_channels 由随机变异器设定为质数（如 37）时，下游 GroupNorm 几乎不
可能满足整除约束。该案例揭示了一类跨算子的隐式参数耦合：单独看每个算子的参数
范围均合法，但算子间的数值关系构成了隐藏的可行域约束。

      传统 Fuzzing 工具通常更关注崩溃现象本身，较难直接给出参数层面的解释。Daikon
引擎通过系统性的双空间对比与模板扫描，将” 这个用例崩溃了” 的二元判断补充为” 该

6.4   语义检索效果评估

      本节评估语义检索模块的召回准确率，验证双路语义检索策略在相似测试用例发现
任务中的有效性。
      语义检索实验从数据集中随机抽取 50 条用例作为查询样本。对每条查询样本，分
别使用系统的双路语义检索接口召回 Top-5 结果，并由人工标注判定每条召回结果是否
与查询样本” 主题相关” （算子类型一致且错误模式相似即判定为相关）。
      在 50 次查询中，Top-1 命中率为 82.0%（41/50），Top-3 召回率为 94.0%（47/50），
Top-5 召回率为 98.0%（49/50）。其中，结构向量路检索在算子拓扑一致的场景下表现
较好，命中率达 92.0%；日志向量路在跨拓扑的相同错误类型检索中也有贡献，命中率
为 78.0%。双路加权融合策略将两者的优势互补，整体 Top-5 召回率较单路结构检索提
升 6.5 个百分点，较单路日志检索提升 12.3 个百分点。该结果说明，在当前 50 条人工
标注查询样本上，结构向量和日志向量能够提供互补的相似性线索；但召回效果仍受标
注标准、查询样本规模和错误类型分布影响。

                                 总结与展望

工作总结

    本文围绕深度学习框架模糊测试中用例冗余、崩溃日志分析成本高、覆盖反馈利
用不足等问题，设计并实现了一套基于语义检索与大语言模型的模糊测试辅助分析平
台。系统采用前后端分离：Java Spring Boot 业务后端运行在 8080 端口，使用 Spring Data
JPA/Hibernate；Python FastAPI 算法后端运行在 5000 端口。case_uid 与业务元数据保
存在 MySQL，Milvus 集合只保存 id 与 embedding，跨库关联依赖 vector_id、parame-
ters.structure_vector_id 和 parameters.log_vector_id。
    在系统功能上，本文完成了自动化日志解析与特征提取、日志文本和脚本结构信息
的双路向量化、相似用例召回、DFS 拓扑回溯的故障辅助定位、Daikon 式候选边界条
件分析、CEI 效能评估与低效用例级联清退、RAG 诊断文本草案生成以及候选补盲脚
本生成。CEI 清退流程依据 MySQL 中保存的向量主键删除 Milvus 对应向量，再删除
MySQL 的 test_case_metadata 记录；由于 MySQL 与 Milvus 不共享事务管理器，该过
程应理解为级联清退和最终一致性维护，并非严格分布式事务保证。前端基于 Vue 3 和
ECharts 图表完成用例管理、覆盖盲区、CEI 结果和多样性分布展示。
    实验结果表明，在当前设置下，CEI 筛选能够减少部分低效用例，并在覆盖贡献基
本保持的同时降低总执行耗时。在 2000 例模拟实验和 1000 例物理跑测中，结果与第
六章覆盖贡献和耗时统计一致。Daikon 候选边界条件分析在五条已知约束上成功召回
全部目标约束，并在一个跨算子隐式整除约束案例中提供了参数层面的排查线索；该
结果不能推广为所有边界条件均可被发现。RAG 模块能够结合历史相似用例和候选约
束生成诊断文本草案，候选补盲脚本也仍需经过执行验证和人工确认。系统性能实验
说明，当前实现可支撑论文实验规模下的检索、分析和展示需求。物理执行部分采用
multiprocessing.spawn 进行进程隔离，结合 Gcov/LCOV 采集覆盖贡献数据，降低单个
异常用例影响主服务的风险。

不足与展望

    当前系统仍存在一些限制。首先，实验数据规模仍然有限。本文主要基于 2000 例
模拟数据、200 例物理对照实验和 1000 例物理跑测进行分析，仍需在更多框架版本、算
           北京航空航天大学毕业设计 (论文)             第   71   页


子类型和更长时间 Fuzzing 任务中验证。其次，CEI 算子权重主要依赖规则化设定，不
能完全等价于真实硬件执行成本；后续可以结合历史执行耗时或 Profiling 结果自适应学
习权重。
  第三，Daikon 候选约束依赖样本质量。如果相似健康样本和崩溃样本数量不足，或
参数分布中存在较多噪声，候选边界条件可能出现漏检或误判。后续可扩展模板库，支
持更多跨算子约束，例如内存步幅、数据对齐、动态形状和稀疏张量相关约束。第四，
RAG 诊断和候选补盲脚本仍需人工复核。LLM 生成内容只能作为诊断文本草案或脚本
草案，不能替代源码级定位、真实复现和回归测试。后续可建立人工评分或模型评分基
准，并完善候选补盲脚本的自动执行、验证和回归测试流程。
  第五，前端可视化和系统扩展性仍有优化空间。当前系统采用 ECharts 完成交互展
示，在更大规模数据下可考虑更高性能的渲染方案、增量式数据加载和增量检索。第六，
跨库一致性仍不是严格分布式事务。当前系统通过主键回填、删除顺序和最终一致性维
护降低不一致风险，后续可以引入补偿日志、任务队列和一致性巡检任务。后续也可扩
展到 TensorFlow、JAX 等 DL 框架，检验本文方法在不同框架和测试流程中的适用性。
