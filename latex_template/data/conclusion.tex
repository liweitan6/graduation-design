\section*{工作总结}

本文围绕深度学习框架模糊测试中的测试产物管理和分析问题，
设计并实现了一套基于语义检索和大语言模型的模糊测试辅助分析系统。
系统采用前后端分离的架构：
Java Spring Boot业务后端运行在8080端口，使用Spring Data JPA/Hibernate；
Python FastAPI算法后端运行在5000端口。
\texttt{case\_uid}和业务元数据保存在MySQL中，
Milvus集合只保存\texttt{id}和\texttt{embedding}，
跨库关联依靠\texttt{vector\_id}、
\texttt{parameters.structure\_vector\_id}和\texttt{parameters.log\_vector\_id}。

在系统功能上，本文完成了测试产物接入和双路向量化流程，
并在此基础上实现了相似用例召回和DFS拓扑回溯的故障辅助定位。
围绕后续的分析，系统还实现了Daikon式候选边界条件分析、
CEI效能评估和低效用例的级联清退、
RAG辅助诊断报告生成和候选补盲测试脚本生成。
CEI清退流程依据MySQL中保存的向量主键删除Milvus中对应的向量，
再删除MySQL的\texttt{test\_case\_metadata}记录；
由于MySQL和Milvus不共享事务管理器，
这一过程应该理解为级联清退和最终一致性维护，
不是严格的分布式事务保证。
前端基于Vue 3和ECharts图表完成主要分析结果的展示。

实验结果表明，在当前设置下，CEI筛选能够减少部分低效用例，
并在覆盖贡献基本保持的同时降低总执行耗时。
在2000例模拟实验和1000例物理跑测中，
结果和第六章的覆盖贡献和耗时统计一致。
Daikon候选边界条件分析在五条已知约束上成功召回了全部目标约束，
并在一个跨算子隐式整除约束的案例中提供了参数层面的排查线索；
这一结果不能推广为所有边界条件都可被发现。
RAG模块可以结合历史相似用例和候选约束生成辅助诊断报告，
候选补盲测试脚本也仍然需要经过执行验证和人工确认。
系统性能实验显示，当前的实现可以满足论文实验规模下的检索、分析和展示需求。
物理执行部分采用\texttt{multiprocessing.spawn}做进程隔离，
结合Gcov/LCOV采集覆盖贡献数据，降低单个异常用例影响主服务的风险。

\section*{不足与展望}

当前系统仍然存在一些限制。首先，实验数据规模仍然有限。
本文主要基于2000例模拟数据、200例物理对照实验和1000例物理跑测做分析，
仍然需要在更多框架版本、算子类型和更长时间的Fuzzing任务中验证。
其次，CEI算子权重主要依靠规则化设定，
不能完全等价于真实的硬件执行成本；
后续可以结合历史的执行耗时或Profiling结果来调整权重。

第三，Daikon候选约束依靠样本质量。
如果相似健康样本和崩溃样本的数量不够，
或参数分布中存在较多噪声，
候选边界条件可能出现漏检或误判。
后续可以扩展模板库，支持更多跨算子的约束类型。
第四，辅助诊断报告和候选补盲测试脚本仍然需要人工复核。
LLM生成的内容只能作为辅助诊断报告或候选脚本草案，
不能替代源码级的定位、真实复现和回归测试。
后续可以建立人工评分或模型评分的基准，
并完善候选补盲测试脚本的执行验证和回归测试流程。

第五，前端可视化和系统扩展仍然有改进空间。
当前系统采用ECharts完成交互展示，
在更大规模数据下可以考虑更适合大数据量展示的渲染方案、增量式数据加载和增量检索。
第六，跨库一致性仍然不是严格的分布式事务。
当前系统通过主键回填、删除顺序和最终一致性维护来降低不一致的风险，
后续可以增加补偿日志、任务队列和一致性巡检任务。
后续也可以扩展到TensorFlow、JAX等DL框架，
检验本文方法在不同框架和测试流程中的适用性。
