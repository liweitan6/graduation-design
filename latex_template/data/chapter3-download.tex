\chapter{系统需求分析与总体设计}

\section{系统需求分析}

\subsection{功能性需求}

根据第一章对DL框架Fuzzing产物管理问题的分析，
本系统围绕测试产物的管理和分析来组织数据处理流程。
进入系统的数据主要来自Fuzzing工具生成的测试脚本、执行日志等测试产物；
经过解析、向量化和统计分析之后，
系统向开发者提供结构化数据和分析结果。
CEI评分、候选边界条件和辅助诊断报告也会在分析页面中展示。
具体的功能性需求包括以下六项。

（1）日志解析与特征提取。
Fuzzing工具持续运行之后会产生大量的原始崩溃日志和测试脚本。
在数据接入阶段，系统读取相关测试产物，
从中抽取错误信息和执行状态等内容。
错误和运行相关的字段会整理成结构化元数据，
用于列表筛选、统计和排序；
日志文本和脚本结构信息则进一步编码为稠密语义向量，
供后续的相似用例检索、崩溃归类和多样性分析使用。

（2）语义检索与故障辅助定位。
开发者拿到新的崩溃日志后，
一般需要先判断它和历史样本中的错误模式是否相近。
系统允许输入自然语言描述、错误片段或崩溃特征，
算法后端把查询内容映射到向量空间，
再召回语义接近的历史用例。
检索结果返回摘要、相似度和相关元数据，
帮助开发者比较当前崩溃和历史样本之间的关系。
在此基础上，故障辅助定位模块结合计算图拓扑关系和错误类型，
给出候选算子节点或参数位置。
这些结果只用来缩小排查范围，
仍需要配合源码阅读、复现实验和人工判断作进一步确认。

（3）测试集多样性监测。
Fuzzing过程容易在局部参数空间或少数算子组合附近反复变异。
多样性监测模块把测试用例的高维语义特征投影为PCA二维坐标，
再由前端基于ECharts散点图展示。
开发者可以结合图中的筛选信息，
观察当前测试集在语义空间中的分布情况。
这一视图主要提供空间分布层面的观察依据，
并不替代覆盖贡献数据和CEI等量化指标。

（4）效能评估与低价值用例清退。
在长时间的Fuzzing任务中，
有些测试用例虽然占用较多执行资源，
但对分支覆盖率或边覆盖率的贡献较低。
效能评估模块读取覆盖贡献数据、算子权重和算子出现次数，
算出每条用例的CEI得分，
再按预设或动态计算得到的CEI阈值标记低收益的冗余用例。
开发者在前端查看单条用例的评分、覆盖贡献和执行成本之后，
可以决定是否执行清退。
清退流程先按MySQL中保存的向量主键删除Milvus里的向量，
再删除MySQL中的元数据记录；
这一过程属于级联清退和最终一致性维护，
不作为严格的事务保证。

（5）候选边界条件分析。
对同一类算子或相似的计算图结构来说，
健康样本和崩溃样本之间往往在参数取值、张量形状或算子属性上存在差异。
候选边界条件分析模块用成功执行样本和崩溃触发样本组成对照组，
把嵌套参数整理成可比较的数值键值对，
再用Daikon式约束推断方法寻找候选边界条件。
前端展示候选约束、关联样本和相关参数，
帮助开发者理解某类崩溃可能和哪些参数范围有关。
这些候选边界条件仍需要配合源码和复现实验复核，
不能直接当作源码级结论。

（6）辅助诊断与算子缺口补盲。
辅助诊断与补盲模块基于RAG方法实现。
在崩溃诊断场景下，
模块把当前崩溃相关的测试上下文组织为提示词，
为LLM生成辅助诊断报告提供项目内的数据依据。
辅助诊断报告包含故障排查参考信息，
可以减少开发者在长日志里手工筛选信息的成本。
对当前测试集中未覆盖或覆盖不足的算子组合，
模块会构造补盲提示词，由LLM生成候选补盲测试脚本。
这类脚本作为后续测试输入的候选材料纳入管理，
仍需要经过执行验证和人工检查。

\subsection{非功能性需求}

（1）批量写入性能。
Fuzzing任务一般以批量或持续的方式产生测试用例，
在批量执行场景下，同一时间段内可能有多条脚本、日志和覆盖记录进入系统。
如果数据导入接口的写入能力不够，
测试结果就会堆积在文件目录或待处理数据中，
影响后续检索和分析的时效性。
数据导入接口需要达到每秒数十条用例的批量写入能力。
实际写入时，算法后端先完成结构文本和日志文本的向量化，
再把语义向量写入Milvus并拿到向量主键，
最后把结构化元数据和向量主键写入MySQL。
对写入失败或部分失败的记录，
系统需要保留重试或异常登记入口，
以减少数据丢失和双库状态不一致的风险。

（2）大规模数据可视化性能。
多样性监测页面需要在浏览器端展示较大规模的二维散点图，
覆盖分析页面还需要展示算子对热力图。
如果每个点或每个网格都用大量DOM节点直接渲染，
缩放、筛选和悬停查看详情时就容易出现卡顿。
因此前端采用ECharts展示散点图和热力图，
由图表组件统一处理展示和交互细节，
避免在页面里维护大量独立的DOM节点。
这一需求不只是界面体验问题，
也关系到研究人员能不能在大量历史用例中观察测试集的分布情况。

（3）底层执行隔离安全性。
系统在采集覆盖率数据或验证候选补盲测试脚本时，
需要实际执行由Fuzzing产生的测试用例。
这类脚本可能触发底层异常或资源占用异常。
如果所有用例都在主服务进程内执行，
单个异常样本就可能导致服务中断，
也会影响其他用户请求和正在进行的数据写入。
为了避免单个异常样本影响主服务，
每条测试用例需要放入独立的子进程中运行，
主进程根据退出状态码和日志捕获拿到执行结果，
还可以配合超时控制减少长时间挂起对服务的影响，
以此隔离底层异常对Web服务本身的影响。

（4）异构数据一致性。
本系统同时用MySQL保存结构化元数据，
用Milvus保存高维语义向量。
一次数据写入一般需要跨越两个不同的存储系统，
删除或清退操作也需要同时处理两边的记录。
由于两类数据库的事务机制和写入接口不同，
系统不能简单靠单库事务覆盖完整流程。
跨库关联由MySQL中的\texttt{case\_uid}和向量主键共同维护：
\texttt{case\_uid}保存在\texttt{test\_case\_metadata}表中，
\texttt{vector\_id}、\texttt{parameters.structure\_vector\_id}
和\texttt{parameters.log\_vector\_id}记录Milvus侧对应向量。
清退时先按这些向量主键删除Milvus记录，
再删除MySQL元数据，
以级联清退和最终一致性维护降低残留数据风险。

\section{系统总体架构设计}

\subsection{前后端分离与混合后端架构}

本系统采用前后端分离的整体架构，
后端进一步拆成业务后端和算法后端两个独立服务。
在工程实现中，这两类任务的依赖和运行方式差异较大：
与用户请求处理相关的任务更适合放在Java业务后端；
与语义分析相关的任务则放在Python算法后端。

前端用Vue 3实现交互界面。
系统界面包含多类管理组件。
Vue 3的组合式API（Composition API）便于把页面状态拆分为可复用的逻辑，
适合用来构建这类管理型单页面应用。
同时，前端承担散点图渲染和交互筛选，
可视化部分基于ECharts组织散点图和热力图，
减少大量DOM节点带来的渲染压力。

业务后端用Java Spring Boot实现，服务端口为8080。
这一层主要负责处理业务请求和关系型数据，并为算法后端提供统一的代理入口。
测试用例元数据、执行状态和CEI得分等查询条件
更适合通过关系型数据库完成组合筛选。
Spring Boot配合Spring Data JPA/Hibernate负责实体映射、Repository接口和事务边界，
复杂的统计查询则通过Repository中的原生SQL聚合语句完成。
所以Java业务后端在系统中承担结构化数据管理和请求路由的职责。

算法后端用Python FastAPI实现，服务端口为5000。
DL领域的算法生态——包括Sentence-BERT语义编码、Milvus客户端和Daikon式约束推断等依赖——
都可以较直接地在Python环境中集成。
把这些计算任务封装成独立服务，
可以避免在Java后端中引入复杂的跨语言模型调用逻辑，
也方便后续调整算法实现。
FastAPI提供异步请求处理和较轻量的API封装方式，
适合承载向量化、检索和分析类接口。
两个后端服务通过HTTP API通信：
前端请求统一发送到Java业务后端的8080端口，
涉及算法计算的请求再由业务后端通过内部代理转发到Python算法后端的5000端口。
这种代理转发模式对前端透明，
可以减少前端同时维护多个服务地址和请求配置的复杂度。

\subsection{异构双库存储架构}

本系统的数据资产包含两种不同形态：
一类是结构化元数据字段，
如\texttt{case\_uid}、执行状态和覆盖贡献数据等；
另一类是由Sentence-BERT编码得到的384维稠密浮点语义向量，
包括日志语义向量和脚本结构信息语义向量。
如果用单一的存储方式管理这两类数据，
要么不太能好好完成多条件结构化查询，
要么不太能支持高维向量的相似度检索。
基于这一数据特点，系统采用MySQL和Milvus协同的异构双库存储架构。

MySQL负责保存结构化元数据和业务查询需要的字段。
例如测试用例的\texttt{case\_uid}、执行状态和覆盖贡献数据等业务字段都保存在MySQL中。
Milvus向量主键、CEI得分和可视化坐标则以表字段或JSON扩展字段的形式保存。
这些数据用于按多种业务条件筛选，
也用于分页、排序和事务性更新。
关系型数据库的索引机制和事务能力更适合处理这类操作。

Milvus负责保存高维语义向量。
日志文本和脚本结构信息经过Sentence-BERT编码之后，
形成固定维度的稠密向量。
这些向量无法通过普通的B+树索引表达语义距离，
需要借助IVF\_FLAT等向量索引做近似最近邻检索。
当前Milvus集合只保存自增主键\texttt{id}和向量字段\texttt{embedding}，
主要用来按查询向量召回相似日志、相似脚本或候选历史用例，
不保存\texttt{case\_uid}等业务标量字段，
也不承担复杂的业务条件查询和事务性数据维护。

两库之间通过MySQL保存的\texttt{case\_uid}和向量主键建立关联。
每条测试用例入库时，
算法后端先把结构向量和日志向量分别写入Milvus集合，
拿到自动生成的向量主键；
随后把\texttt{case\_uid}写入MySQL主记录，
把结构向量主键写入\texttt{vector\_id}字段，
并在\texttt{parameters} JSON字段中保存
\texttt{structure\_vector\_id}和\texttt{log\_vector\_id}。
查询时，结构化筛选以MySQL为主；
语义检索先由Milvus返回相似向量主键，
算法后端再按MySQL中保存的\texttt{vector\_id}、
\texttt{parameters.structure\_vector\_id}或
\texttt{parameters.log\_vector\_id}反查完整元数据。
这种分工让MySQL负责业务记录和跨库关联，
Milvus专注于向量相似度召回。

\subsection{系统技术栈选型说明}

基于上述架构决策，本系统各层技术栈选型如下。
技术选型主要围绕系统已有的数据形态、算法依赖和交互需求展开，
而不是简单比较不同框架的通用优劣。

前端层采用Vue 3框架配合TypeScript开发。
本系统前端包含较多状态联动，
包括测试用例筛选条件、分页结果和详情面板等前端状态。
Vue 3的组合式API适合把这些逻辑拆成相对独立的状态单元。
TypeScript则用来约束接口返回的数据结构，
减少前后端字段变更带来的运行时错误。
UI组件库选用Element Plus，
它的表格、表单和对话框等组件能够覆盖管理系统的基础交互需求。
散点图和热力图渲染基于ECharts完成，
以满足管理界面中筛选、悬停和局部查看等交互需求。

业务后端层采用Java 17和Spring Boot 3框架。
这一层面对的是结构化业务数据和前端请求组织，
主要处理常规的业务查询和异常响应。
数据访问层使用Spring Data JPA/Hibernate作为ORM工具，
通过实体类映射\texttt{test\_case\_metadata}表，
常规查询由Repository接口派生方法完成，
统计类的结果则使用Repository中的原生SQL聚合查询。
数据库连接池采用HikariCP，
用来提升多请求访问时的连接复用效率。
这一层同时作为前端和算法后端之间的代理，
减少前端直接调用多个后端服务的复杂度。

算法后端层采用Python 3.11和FastAPI框架。
这一层主要依赖语义编码、向量检索和数据处理相关的组件，
支撑向量化、检索和数据处理等算法任务。
FastAPI用来把这些算法能力封装成RESTful接口，
便于业务后端按需调用。

存储层中，MySQL 8.0负责结构化元数据管理，
Milvus 2.3.x负责高维语义向量的索引和检索。
两者都部署在本地服务器上，
通过各自的SDK和后端服务通信。
MySQL更适合事务性记录和条件查询，
Milvus更适合向量相似度召回。
二者通过MySQL保存的\texttt{case\_uid}和向量主键保持数据关联，
共同为系统的检索、分析和展示功能提供支撑。

\section{系统功能模块划分与数据流}

本系统的功能模块按数据接入、检索分析和辅助生成等职责划分成若干子系统。
其中与语义分析和候选生成相关的算法接口主要封装在Python算法后端；
与业务查询和前端组织相关的功能则由Java业务后端承担。
业务后端根据用户请求调用自身的数据接口或转发到算法后端，
再把处理结果统一组织后返回前端。
在实际开发中，这种划分让不同处理环节保留了清晰的边界，
也方便分别调试业务逻辑和算法逻辑。

系统的整体数据流转路径如下。
原始Fuzzing产物经数据导入接口进入系统，
其中包括测试产物以及运行环境信息。
日志解析模块首先对原始文本做清洗，
过滤冗余调用帧，
提取错误和运行相关的字段。
对能够表示计算图结构的脚本信息，
系统会进一步整理计算图结构信息，
作为后续向量化和辅助定位的输入。

解析结果随后进入向量化环节。
日志文本会生成日志语义向量；
脚本结构信息会生成结构语义向量。
两路语义向量先写入Milvus并返回主键，
结构化元数据、\texttt{case\_uid}和向量主键随后写入MySQL。
写入过程中需要记录两库写入状态，
并在部分失败时执行重试或异常登记，
以降低后续查询中出现孤立记录的风险。

入库完成之后，下游分析模块可以按需启动。
语义检索模块接收用户查询或目标用例标识，
在Milvus中召回相似向量主键，
再按MySQL中保存的\texttt{vector\_id}、
\texttt{parameters.structure\_vector\_id}或
\texttt{parameters.log\_vector\_id}反查相关元数据，然后返回前端。
故障辅助定位逻辑结合错误类型和计算图拓扑关系，
输出可能相关的算子节点或参数位置。
多样性监测模块在用户请求或数据批量更新后读取测试用例向量，
执行PCA降维并输出二维坐标，
供前端散点图渲染。

效能评估模块读取CEI计算所需的信息，
算出各用例的CEI得分，
再按预设或动态计算得到的CEI阈值标记低收益的冗余用例。
当用户确认执行清退时，
系统按MySQL中保存的向量主键删除Milvus里的向量记录，
再删除MySQL中的元数据记录，
通过级联清退和最终一致性维护减少双库残留不一致的风险。
候选边界条件分析模块接收到目标崩溃用例后，
先召回相似的健康样本和崩溃样本，
再对参数键值对作比较，
输出候选边界条件和对应的样本依据。
辅助诊断与补盲模块则把已有的分析结果和测试上下文整合起来构建RAG上下文，
调用LLM生成辅助诊断报告或候选补盲测试脚本。
