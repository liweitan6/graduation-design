\chapter{系统需求分析与总体设计}

\section{系统需求分析}

\subsection{功能性需求}

根据第一章对DL框架Fuzzing产物管理痛点的分析，
本系统需要满足以下六项核心功能性需求。

（1）自动化日志解析与特征提取。
系统应能接收Fuzzing工具产出的原始崩溃日志与测试脚本，
自动从中提取结构化元数据字段（如错误类型、触发算子、张量维度配置等），
并将非结构化文本编码为稠密语义向量，以供后续检索与分析使用。

（2）语义检索与故障溯源。
系统应支持用户以自然语言描述或崩溃特征片段为查询条件，
在向量空间中召回语义相近的历史用例。
同时，系统应提供基于计算图拓扑关系的故障溯源能力，
辅助用户定位底层算子级别的缺陷根因。

（3）测试集多样性监测。
系统应将全量测试用例的高维语义特征降维至二维平面并进行可视化展示，
供研究人员监控当前测试集在语义空间中的分布密度与覆盖广度。

（4）效能评估与低价值用例清退。
系统应建立覆盖率与成本相结合的效能量化指标，
识别对全局覆盖率贡献不足但执行成本偏高的冗余用例，
并支持按阈值自动执行跨数据库的级联清理操作。

（5）故障边界定界。
系统应能基于成功与崩溃两组对照样本的参数分布，
自动推断区分正常执行与异常触发的数学边界条件，
以可视化方式呈现算子输入参数的约束红线。

（6）智能诊断与算子缺口补盲。
系统应集成RAG机制，在崩溃诊断场景中为LLM注入历史相似案例的上下文信息。
同时，系统应能识别当前测试集尚未覆盖的算子组合缺口，
驱动LLM定向生成补充测试脚本并纳入管理。

\subsection{非功能性需求}

（1）高并发写入性能。
在Fuzzing工具持续产出的工作场景下，
系统的数据摄入接口应支持每秒数十条用例的并发写入，
结构化元数据与语义向量的双库落盘过程不应出现阻塞或数据丢失。

（2）大规模数据可视化性能。
多样性监测页面需在浏览器端渲染万级规模的二维散点图。
系统应在主流硬件配置下维持流畅的交互帧率。

（3）底层执行隔离安全性。
系统在执行测试用例以采集覆盖率数据时，
需要将每条用例置于独立进程中运行，
防止段错误或线程死锁等异常传播至主服务进程。

（4）异构数据一致性。
结构化元数据与语义向量分别存储于两个异构数据库中。
在执行写入或删除操作时，系统应保证两库之间的数据一致性，
避免出现单库残留的脏数据。

\section{系统总体架构设计}

\subsection{前后端分离与混合后端架构}

本系统采用前后端分离的整体架构，后端进一步拆分为业务后端与算法后端两个独立服务。
这一设计决策基于以下工程考量。

DL领域的核心算法生态——包括语义编码模型的推理、向量数据库的客户端驱动以及科学计算库——
均以Python作为第一语言支持。
将向量化、降维分析、Daikon边界推断等计算密集型任务封装为独立的Python服务，
可直接调用PyTorch、scikit-learn与Milvus的Python SDK，避免跨语言绑定带来的兼容性成本。
该服务基于FastAPI框架构建，
通过异步协程（asyncio）机制处理并发的应用程序编程接口（Application Programming Interface, API）请求。

与此同时，面向前端的业务逻辑——如用户鉴权、分页查询、关系型数据的CRUD操作——
对运行时类型安全与成熟的对象关系映射（Object-Relational Mapping, ORM）工具链有明确需求。
Java生态在这一领域具有长期积累：
Spring Boot框架提供了完备的依赖注入与事务管理机制，
MyBatis支持灵活的SQL映射与动态查询构建。
因此，业务后端采用Java Spring Boot实现，负责接收前端请求并路由至自身或算法后端。

两个后端服务通过HTTP API进行通信。
前端所有请求统一发送至Java业务后端的8081端口，
涉及算法计算的请求由业务后端通过内部代理转发至Python算法后端的8000端口。
这种代理转发模式对前端透明，降低了前端的集成复杂度。

\subsection{异构双库存储架构}

本系统的数据资产包含两种截然不同的形态：
一是结构化的元数据字段（如错误类型枚举、算子名称、张量维度、时间戳、CEI得分等），
二是由Sentence-BERT编码产出的384维稠密浮点语义向量。
对这两类数据采用单一存储引擎进行统一管理，
将不可避免地在查询性能或功能适配性上产生瓶颈。

关系型数据库MySQL适用于结构化元数据的持久化存储。
其成熟的索引机制（B+树索引）能够支撑基于枚举字段、时间范围或数值区间的条件查询与排序操作。
但MySQL不具备对高维浮点向量执行ANN检索的能力，
强行将384维向量序列化为BLOB字段存储将完全丧失向量空间中的距离语义。

Milvus作为专用的向量检索引擎，
支持在IVF\_FLAT索引上对百万级384维向量执行亚秒级余弦相似度查询。
但Milvus的标量过滤能力有限，不适合承载复杂的多条件组合查询与事务性写入。

基于上述分析，本系统采用MySQL与Milvus协同的异构双库存储架构。
每条测试用例在入库时被分配一个全局唯一标识符（case\_uid），
该标识符同时写入MySQL的主键列与Milvus向量记录的标量附属字段，
作为两库之间的关联纽带。
查询时，系统根据请求类型选择主查询库：
结构化条件筛选由MySQL处理，语义相似度检索由Milvus处理，
跨库结果通过case\_uid执行关联拼接后返回前端。

\subsection{系统技术栈选型说明}

基于上述架构决策，本系统各层技术栈选型如下。

前端层采用Vue 3框架配合TypeScript进行开发。
Vue 3的组合式API（Composition API）适用于构建包含复杂状态逻辑的单页面应用。
UI组件库选用Element Plus，其表格、表单与对话框组件可覆盖管理系统的基础交互需求。
大规模散点图的渲染依赖WebGL加速方案，以满足万级数据点的实时交互要求。

业务后端层采用Java 17与Spring Boot 3框架。
数据访问层使用MyBatis作为ORM工具，
通过XML映射文件实现动态SQL的灵活拼装，便于适配多条件组合查询场景。
数据库连接池采用HikariCP，保障高并发下的连接复用效率。

算法后端层采用Python 3.11与FastAPI框架。
FastAPI原生支持异步请求处理与自动化API文档生成，
适用于包装计算密集型算法为RESTful服务。
该层依赖的核心Python库包括：
sentence-transformers（语义编码）、pymilvus（Milvus客户端）、
scikit-learn（PCA降维与统计计算）以及httpx（异步HTTP客户端）。

存储层中，MySQL 8.0负责结构化元数据管理，
Milvus 2.3.x负责高维语义向量的索引与检索。
两者均部署于本地服务器，通过各自的SDK与后端服务通信。

\section{系统功能模块划分与数据流}

本系统的功能模块按照职责划分为六个子系统：
数据摄入与解析模块、语义检索与故障溯源模块、多样性监测模块、
效能评估与清退模块、边界定界模块以及智能诊断与补盲模块。
各模块均以独立的API端点形式封装于算法后端中，
由业务后端统一调度并向前端暴露服务。

系统的端到端数据流转路径如下。
原始Fuzzing产物（崩溃日志与测试脚本）经数据摄入接口进入系统后，
首先由日志解析模块提取结构化元数据并过滤冗余调用帧。
解析结果随后进入双路向量化环节，
分别生成报错堆栈语义向量与图拓扑序列向量。
结构化元数据落盘至MySQL，语义向量落盘至Milvus，两库通过case\_uid保持关联。

入库完成后，下游分析模块可按需启动：
语义检索模块接受自然语言查询，在Milvus中召回相似用例并从MySQL补全元数据后返回；
多样性监测模块定期拉取全量向量并执行PCA降维，输出二维坐标供前端散点图渲染；
效能评估模块读取覆盖率增量数据与算子权重信息，计算各用例的CEI得分，
并根据动态阈值标记低价值用例；
边界定界模块接收目标崩溃用例，召回对照样本组后输出参数边界条件；
智能诊断模块整合检索结果构建RAG上下文，调用LLM生成诊断报告或补盲测试脚本。
