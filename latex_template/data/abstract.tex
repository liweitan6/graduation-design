\begin{cabstract}

深度学习框架（如PyTorch、TensorFlow）是模型训练和推理的基础软件，
框架底层算子的实现会影响上层程序的运行结果。
模糊测试（Fuzzing）可以产生大量包含边界参数和算子组合的测试脚本，
但实际管理这些产物时也会遇到一些问题：
相似或低收益用例会增加执行开销，
崩溃日志一般较长，里面混合了多层调用信息，
相关测试记录如果分散存放，
后续筛选和故障分析就需要较多人工整理。

本文设计并实现了一套结合语义检索和大语言模型（LLM）的模糊测试辅助分析系统。
系统采用前后端分离架构，前端用Vue 3做交互界面，
算法服务由Python FastAPI承载，业务数据由Java Spring Boot管理，
结构化元数据和高维语义向量分别保存在MySQL和Milvus里。
系统把各类测试产物放进同一条数据流程中，
为后续的用例筛选和故障辅助分析提供基础。

为了减少低收益用例的执行开销，本文构建了覆盖效能指数（Coverage Efficiency Index, CEI）模型。
CEI同时考虑单个测试用例的分支覆盖贡献和加权计算成本，
再通过动态百分位阈值筛选低收益用例。
在2000例的模拟实验中，该方法在覆盖率损失不足4\%的情况下把执行耗时缩减了60\%；
在1000例的真实物理实验中，Gcov/LCOV探针采集到的物理命中行数仅下降0.005\%，
总执行耗时缩减34.5\%。

针对崩溃故障辅助定位，系统设计了基于相似样本对比的Daikon候选边界条件分析模块，
把语义相似用例召回和参数约束推断结合起来寻找候选边界条件。
该模块先召回与目标崩溃用例算子拓扑相同的健康组和崩溃组样本，
把嵌套的张量参数展平成一维数值键值对，
再用预设的约束模板扫描候选不变量。
只有当某条数学约束在健康组样本中全部成立、在崩溃组样本中全部不成立时，
系统才把它标记为候选边界条件。
实验结果显示，该模块在五条已知约束上的召回率达到100\%，
并在真实数据中给出了跨算子的隐式整除候选约束，
可以为崩溃分析提供参数层面的排查依据。

系统进一步利用检索增强生成（Retrieval-Augmented Generation, RAG）方法，
把相似用例上下文和Daikon推断出的候选约束写入结构化提示词模板，
让LLM生成包含相关用例和复现线索的辅助诊断报告，
同时输出候选补盲测试脚本。
这些结果主要为开发者提供相似案例和复现方向参考，
仍需要配合人工复核使用。

此外，系统在物理执行环节使用基于\texttt{multiprocessing.spawn}的子进程隔离方案。
当测试脚本触发底层C++段错误或断言失败时，
主进程根据退出状态码识别异常并回收子进程，
从而减少单个异常用例影响服务进程的可能。

\end{cabstract}

\begin{eabstract}

Deep learning frameworks such as PyTorch and TensorFlow are basic software
used in model training and inference.
The implementation of their low-level operators
affects the execution results of upper-level programs.
Fuzz testing can produce many test scripts containing boundary parameters
and operator combinations,
but these outputs are not easy to manage in practice.
Similar or low-benefit cases increase execution cost,
crash logs often contain long cross-layer stack traces,
and coverage contribution data, script structures, and runtime status
require additional organization before later analysis.

This thesis implements a platform for managing and analyzing fuzzing outputs.
The system follows a decoupled front-end/back-end architecture:
Vue~3 is used for the interactive front end,
Python FastAPI provides the algorithmic service,
Java Spring Boot manages business data,
and MySQL with Milvus stores structured metadata
and high-dimensional semantic vectors.
The platform organizes test cases, coverage information, and crash logs
in one analysis workflow,
supporting test case filtering, fault localization assistance,
candidate boundary-condition analysis, and auxiliary diagnostic reports.

To reduce the execution cost of low-benefit cases,
this thesis builds a Coverage Efficiency Index (CEI) model.
CEI considers both the branch coverage contribution
and the weighted computational cost of each test case,
and filters low-benefit cases using a dynamic percentile threshold.
In a simulated experiment with 2{,}000 cases,
the method reduced execution time by 60\% with less than 4\% coverage loss.
In a physical experiment with 1{,}000 cases instrumented by Gcov/LCOV probes,
the number of physically hit lines decreased by only 0.005\%,
while the total execution time was reduced by 34.5\%.

For crash analysis assistance,
the system implements a Daikon-based candidate boundary-condition analysis module
using similar-sample comparison,
which combines semantic retrieval of similar cases
with parameter-constraint inference to identify candidate boundary conditions.
The module retrieves healthy and crashed samples
that share the same operator topology as the target crash case,
flattens nested tensor parameters into one-dimensional numerical key-value pairs,
and scans candidate invariants using predefined constraint templates.
A mathematical constraint is reported as a candidate boundary condition
only when it is satisfied by all healthy samples
and violated by all crashed samples.
The module reaches a 100\% recall rate on five known constraints
and also reports candidate boundary conditions in real data,
including implicit divisibility constraints across operators.
These results provide parameter-level clues for crash analysis,
but they still need source-code inspection and reproduction experiments.

Retrieval-Augmented Generation (RAG) is then used
to organize retrieved cases and inferred constraints for LLM-based analysis.
The retrieved similar-case contexts and Daikon-inferred candidate constraints
are inserted into a structured prompt template,
so that the LLM can generate auxiliary diagnostic reports containing related cases,
suspicious constraints, and reproduction clues,
as well as candidate gap-filling test scripts.
These outputs are used as references for developers
and still require manual review and execution validation.

The system also implements a process-level isolated execution module
based on \texttt{multiprocessing.spawn}.
By capturing child process exit codes,
this module records low-level C++ segmentation faults and assertion failures
without running the test script in the main service process.

\end{eabstract}
