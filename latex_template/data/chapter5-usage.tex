各层之间的通信链路安排如下。
前端运行于Vite开发服务器（默认端口5173），
通过Axios HTTP客户端分别向两个后端发起请求。
基础用例管理类请求（分页查询、状态筛选、详情获取）
指向Java Spring Boot后端（端口8080）；
所有涉及向量计算、语义检索、RAG推理与Daikon定界的分析类请求
指向Python FastAPI后端（端口5000）。
Java后端对部分分析类请求实现了代理转发机制：
当前端调用\texttt{/api/search}或\texttt{/api/ask}时，
Java后端将请求体原样转发至Python后端对应路由，
并将响应透传回前端。
该设计使前端仅需维护单一的基础URL配置，
同时保留了Python后端独立扩展的灵活性。

底层物理执行环境部署于WSL2内的Ubuntu 22.04子系统。
该子系统中安装了携带Gcov探针的PyTorch编译版本，
以及独立的Python虚拟环境\texttt{venv\_pytorch}。
物理执行引擎通过\texttt{multiprocessing.spawn}启动隔离进程，
在WSL环境下调用插桩后的PyTorch库执行测试脚本并采集覆盖率数据。

\section{后端核心模块实现}

\subsection{Python FastAPI算法后端实现}

该服务以单文件形式组织（\texttt{data\_service.py}），
启动时通过Uvicorn ASGI服务器监听5000端口。
服务初始化阶段依次完成以下资源加载：
（1）建立Milvus连接并加载两个向量集合至内存；
（2）加载预训练的Sentence-BERT模型至GPU（若可用）或CPU；
（3）验证MySQL连接池可用性。

该后端共注册了12条API路由，按功能划分为五组。

以下以核心接口\texttt{/api/analysis/}\allowbreak\texttt{fault-boundary}
为例，详细说明请求体与响应体的JSON结构设计。

该接口的请求体仅包含一个字段：

\texttt{\{"failed\_case\_uid": "<string>"\}}

其中\texttt{failed\_case\_uid}为目标崩溃用例的全局唯一标识符。
接口内部执行以下处理链路：
（1）从MySQL查询目标用例的完整元数据与向量主键；
（2）以向量主键在Milvus中提取结构特征向量；
（3）在Milvus中执行余弦相似度检索，召回20个最近邻样本；
（4）将近邻样本按执行状态分为健康组（最多10个）与崩溃组（最多10个）；
（5）对健康组与崩溃组执行参数展平与张量Diff对比；
（6）调用Daikon定界引擎推断被违反的数学不变量；
（7）将Diff结果与数学红线注入LLM提示词，生成自然语言诊断。

响应体的JSON结构包含五个顶层字段，嵌套层级如下：

（1）\texttt{failed\_case}对象。
包含\texttt{uid}（崩溃用例标识符，字符串类型）
与\texttt{error}（原始错误文本，字符串类型）两个子字段。

（2）\texttt{successful\_case}对象。
包含\texttt{uid}（最近邻健康用例标识符）一个子字段。

（3）\texttt{diff}对象。
包含\texttt{successful\_values}与\texttt{failed\_values}两个子字段，
均为键值对字典，键为展平后的参数路径（如\texttt{layers.conv1.}%
\allowbreak\texttt{kernel\_size}），
值为对应的数值。
该字段供前端渲染双侧参数对比视图。

（4）\texttt{daikon\_violations}数组。
每个元素为一条字符串形式的数学约束表达式，
例如\texttt{"layers.conv1.kernel\_size}%
\allowbreak\texttt{ + 2 * layers.conv1.padding[0]}%
\allowbreak\texttt{ >= layers.conv1.input\_size"}。
数组按变量数量降序排列，多变量约束排在前面。

（5）\texttt{boundary\_rule}字符串。
LLM基于上述所有上下文生成的自然语言诊断摘要。

数据摄入接口\texttt{/api/ingest}采用POST方法，
请求体为TestCase对象的JSON数组。
每个TestCase对象包含以下字段：
\texttt{case\_uid}（字符串，全局唯一标识）、
\texttt{status}（字符串，执行状态枚举）、
\texttt{edge\_coverage}（整数，边覆盖数）、
\texttt{execution\_time}（浮点数，执行耗时秒）、
\texttt{error\_message}（字符串或空值，错误文本）、
\texttt{model\_structure}（嵌套字典，包含\texttt{layers}列表、
\texttt{connections}连接链、\texttt{depth}深度与\texttt{operators}算子列表四个键）。
响应体包含\texttt{success}（成功摄入数量）、
\texttt{ids}（MySQL主键列表）与\texttt{message}（状态描述）三个字段。

语义检索接口\texttt{/api/search}的请求体包含单一字段
\texttt{query}（用户自然语言查询文本）。
接口内部首先对查询文本执行意图分类，
判断用户意图为结构检索、日志检索或混合检索，
并据此动态调节双路检索权重。
响应体包含\texttt{results}数组与\texttt{query\_info}对象。
\texttt{results}数组中每个元素包含用例的完整元数据与相似度得分；
\texttt{query\_info}对象记录检索参数，
包括\texttt{intent}（意图分类结果）、
\texttt{structure\_weight}与\texttt{log\_weight}（双路权重）。

\subsection{Java Spring Boot业务后端实现}

Java Spring Boot后端承载基础的业务数据管理功能，
监听8080端口并通过MyBatis ORM框架与MySQL交互。
该后端的核心职责包括三项：
用例的分页查询与条件筛选、统计数据的聚合计算、
以及对Python后端分析类接口的代理转发。

用例分页查询接口\texttt{/api/cases}支持三个查询参数：
\texttt{page}（页码，默认0）、\texttt{size}（每页数量，默认10）、
\texttt{status}（可选的状态过滤条件）。
MyBatis Mapper层将这些参数映射为带有\texttt{LIMIT}与\texttt{OFFSET}子句的SQL语句。
当\texttt{status}参数非空时，Mapper动态拼接\texttt{WHERE status = ?}条件。
响应体采用Spring Data风格的分页封装，
包含\texttt{content}（当前页用例数组）、\texttt{totalElements}（总记录数）、
\texttt{totalPages}（总页数）与\texttt{number}（当前页码）四个字段。

统计接口组包含三条路由：
\texttt{/api/statistics/overview}返回全局用例总数、崩溃率与平均执行耗时；
\texttt{/api/statistics/status}返回各状态类别的用例计数；
\texttt{/api/statistics/errors}返回错误类型排行榜。
三条路由均通过MyBatis执行\texttt{GROUP BY}聚合查询，
将结果封装为前端图表组件可直接消费的JSON格式。

代理转发机制的实现基于Spring的\texttt{RestTemplate}组件。
Java后端在Controller层注册了与Python后端同名的路由映射
（如\texttt{/api/search}、\texttt{/api/ask}）。
当请求到达时，Controller将请求体序列化后通过\texttt{RestTemplate}
转发至\texttt{http://localhost:5000}对应路径，
并将Python后端的响应体原样返回给前端。
转发超时设置为60秒，以适应LLM推理可能产生的较长延迟。

两个集合在服务启动时执行\texttt{collection.load()}操作，
将索引结构与向量数据加载至Milvus的查询节点内存中。
该操作确保后续的检索请求无需触发磁盘I/O，
在十万级向量规模下可保障单次检索延迟低于10毫秒。

跨库数据一致性通过写入顺序与补偿机制共同保障。
数据摄入时严格遵守"先Milvus后MySQL"的写入顺序：
引擎先将双路向量分别插入两个Milvus集合并获取自动生成的主键，
随后将主键嵌入MySQL记录的\texttt{parameters} JSON字段。
若Milvus写入成功但MySQL写入失败，
系统将失败记录重新推入缓冲队列并记录告警日志。
数据删除时则遵守"先查MySQL后删Milvus再删MySQL"的三步顺序，
通过内存级互斥标记防止删除期间的脏读。

\section{前端界面实现与功能展示}

本系统前端基于Vue 3组合式API（Composition API）与TypeScript构建，
采用Element Plus组件库提供统一的交互控件。
整体视觉风格为深色主题（背景色\texttt{\#0f172a}）配合靛蓝主色调（\texttt{\#6366f1}），
呈现极客风格的玻璃拟态（Glassmorphism）效果。
所有页面组件在\texttt{onMounted}生命周期钩子中触发数据拉取，
通过Axios实例向对应后端发起HTTP请求并将响应数据绑定至响应式状态变量。

\subsection{系统总览仪表盘}

系统总览仪表盘（Dashboard）作为用户进入系统后的首屏，
提供全局测试状态的聚合视图。
页面顶部排列四张统计卡片，分别展示累计测试用例数、测试通过率、异常崩溃总数与平均执行耗时。
卡片数据通过调用\texttt{/api/statistics/overview}接口获取，
接口返回的JSON对象中各字段直接绑定至卡片的数值插槽。

页面中部为测试状态分布环形图。
该图基于\texttt{vue-echarts}组件封装，
在\texttt{onMounted}钩子中调用\texttt{getStatisticsStatus()}函数拉取状态分布数据。
接口返回的数组格式为\texttt{[\{name: "Success", value: N\}, ...]}，
该数组直接赋值给ECharts配置对象的\texttt{series[0].data}属性。
ECharts内部基于Canvas 2D上下文渲染环形区域，
每个弧段的弧度由\texttt{value}字段占总和的比例决定。
环形图以通过与失败两种状态着色，支持鼠标悬停高亮。

页面底部为最近测试用例列表。
通过\texttt{getRecentCases(10)}拉取最近入库记录，

状态标签通过Element Plus的\texttt{el-tag}组件渲染，
不同状态映射不同的颜色主题（Success为绿色、Crash为红色、Timeout为橙色）。

\subsection{用例管理与智能语义检索界面}

智能语义检索页面（Search）提供自然语言查询入口。

\subsection{故障边界对比与Daikon定界展示界面}

关联分析页面（Correlations）承载故障边界定界引擎的可视化交互。
页面顶部提供崩溃用例选择器，用户选中目标崩溃用例后，
前端调用\texttt{analyzeFaultBoundary}\allowbreak\texttt{(failedCaseUid)}
发起分析请求。

请求返回后，页面呈现三个功能区域。
第一区域为双侧参数对比视图。
左侧面板渲染崩溃用例的展平参数键值对列表，
右侧面板渲染最近邻健康用例的展平参数列表。
两侧相同键的行水平对齐，数值差异的行以红色高亮标注。
该对比数据来源于响应体中\texttt{diff.failed\_values}
与\texttt{diff.successful\_values}两个字典。
前端通过JavaScript的\texttt{Object.keys()}提取键集合并排序，
逐键比对数值是否一致以决定是否施加高亮样式类。

第二区域为数学红线告警面板。

\section{底层物理执行沙箱实现}

本节描述系统中物理层面的测试执行基础设施。

其核心任务为：在携带覆盖率探针的PyTorch编译版本上执行测试脚本，
采集C++层的分支命中数据，
并以进程级隔离机制防止测试脚本的崩溃向主服务传播。

\subsection{Gcov/LCOV探针的编译挂载流程}

Gcov探针的工作原理基于编译期插桩与运行期计数两阶段协作。
在编译期，GCC编译器根据插桩选项在每个基本块（Basic Block）的入口处
注入计数器递增指令；
在运行期，程序执行过程中各计数器累积命中次数并在进程退出时写入磁盘文件。

为对PyTorch底层C++源码启用Gcov插桩，
需在WSL环境中修改编译选项。
具体方式为设置如下环境变量后重新触发CMake配置与编译：

\begin{verbatim}
export CXXFLAGS="-fprofile-arcs -ftest-coverage"
export LDFLAGS="-lgcov --coverage"
\end{verbatim}

其中\texttt{-fprofile-arcs}指令使编译器在每个分支跳转点插入弧计数器，
\texttt{-ftest-coverage}指令使编译器为每个编译单元生成\texttt{.gcno}文件。
链接阶段的\texttt{-lgcov}参数将Gcov运行时库静态链接至目标二进制。

编译完成后，每个\texttt{.cpp}源文件对应产生一个\texttt{.gcno}文件。
该文件记录源码的控制流图拓扑——
包括基本块编号、块间有向边关系与源码行号映射。
\texttt{.gcno}文件在编译时一次性生成，后续不再变化。

当携带探针的PyTorch库被测试脚本加载并执行后，
运行时库在进程正常退出（或收到\texttt{SIGTERM}信号）时，
将每个弧计数器的累积值写入与\texttt{.gcno}同目录的\texttt{.gcda}文件。
\texttt{.gcda}文件记录每条边在本次执行中被穿越的次数。
多次执行产生的\texttt{.gcda}数据会累积叠加。

覆盖率数据的提取通过LCOV命令行工具完成。
执行流程分为三步：

（1）基线采集。在任何测试执行之前，运行
\texttt{lcov --capture --initial}%
\allowbreak\texttt{ --directory /path/to/pytorch/build}%
\allowbreak\texttt{ --output-file baseline.info}
生成零计数基线文件。

（2）增量采集。测试脚本执行完毕后，运行
\texttt{lcov --capture}%
\allowbreak\texttt{ --directory /path/to/pytorch/build}%
\allowbreak\texttt{ --output-file test.info}
采集含有命中计数的覆盖率快照。

（3）差分计算。通过
\texttt{lcov --remove test.info}%
\allowbreak\texttt{ '/usr/*' --output-file filtered.info}
过滤系统头文件贡献后，
再利用\texttt{genhtml filtered.info}%
\allowbreak\texttt{ --output-directory coverage\_html}
生成可视化HTML报告。

该报告中的"行命中数"即为本系统CEI模型中\texttt{edge\_coverage}字段的物理数据来源。
每次物理执行完成后，Python后端解析\texttt{.info}文件提取总命中行数，
并将其回写至MySQL对应用例记录。

\subsection{基于multiprocessing.spawn的隔离执行引擎}

测试脚本的执行环境面临两类致命风险：
其一，脚本可能触发底层C++算子的段错误（Segmentation Fault），
导致整个进程地址空间被操作系统回收；
其二，PyTorch内部依赖的OpenMP线程池在\texttt{fork}模式下
可能继承父进程的锁状态，造成不可恢复的死锁。
若测试脚本与主Web服务共享进程空间，任一风险均将导致后端服务宕机。

为彻底消除上述风险，本系统采用\texttt{multiprocessing}模块的\texttt{spawn}启动方式。
与默认的\texttt{fork}模式不同，\texttt{spawn}不复制父进程的内存页表，
而是启动一个全新的Python解释器进程，
从零开始导入目标模块并执行指定函数。
该机制确保子进程持有完全独立的地址空间、线程池与GPU上下文。

启动方式通过在服务初始化阶段调用
\texttt{multiprocessing.set\_start\_method}%
\allowbreak\texttt{('spawn', force=True)}全局设定。
\texttt{force=True}参数覆盖可能已被第三方库设置的启动模式。
此后所有通过\texttt{multiprocessing.Process}创建的子进程
均以\texttt{spawn}方式启动。

每个测试脚本的执行流程如下。
主进程构造\texttt{Process}对象，将执行函数与参数序列化后传入：

\begin{verbatim}
p = multiprocessing.Process(
    target=_execute_wrapper,
    args=(case_uid, input_shape_str, operators_str)
)
p.start()
p.join(timeout=15)
\end{verbatim}

\texttt{p.join(timeout=15)}使主进程阻塞等待子进程完成，
最长等待15秒。若子进程在超时时限内未退出，
主进程调用\texttt{p.terminate()}发送\texttt{SIGTERM}信号强制终止子进程，
随后调用\texttt{p.join()}等待资源回收完成。
该超时机制防止因死循环或无限递归导致的资源泄漏。

子进程退出后，主进程通过\texttt{p.exitcode}属性获取退出状态码。
退出状态码的语义遵循POSIX标准：
值为0表示正常退出；
正整数表示Python层未捕获异常导致的退出（如值1为通用错误）；
负整数表示子进程被操作系统信号终止，其绝对值为信号编号。

当底层C++算子因内存越界触发段错误时，
Linux内核向子进程发送Signal 11（\texttt{SIGSEGV}）。
由于子进程未注册该信号的处理函数，
默认行为为立即终止进程并产生核心转储（若已启用）。
此时主进程观察到\texttt{p.exitcode == -11}。

主进程根据退出状态码执行分类处理：

\begin{verbatim}
if p.exitcode == 0:
    status = "SUCCESS"
elif p.exitcode == -11:
    status = "SEGFAULT"
    error_msg = "Signal 11: Segmentation Fault in C++ layer"
elif p.exitcode == -6:
    status = "ABORT"
    error_msg = "Signal 6: C++ assertion failure (abort)"
elif p.is_alive():
    p.terminate()
    p.join()
    status = "TIMEOUT"
else:
    status = "CRASH"
    error_msg = f"Exit code: {p.exitcode}"
\end{verbatim}

Signal 11（段错误）与Signal 6（\texttt{SIGABRT}，断言失败）
是PyTorch底层算子在非法参数组合下最常触发的两类信号。
本系统将两者均转换为结构化的错误状态记录，
写入MySQL的\texttt{status}与\texttt{error\_message}字段，
供后续的DFS故障溯源与Daikon定界引擎使用。

该隔离机制的核心保障在于：
无论子进程以何种方式终止——正常退出、异常退出或被信号杀死——
主进程的地址空间、线程池与网络连接均不受影响。
FastAPI的ASGI事件循环持续响应其他HTTP请求，
系统可用性不因单个测试脚本的C++层崩溃而中断。

OpenMP线程死锁的防护同样依赖\texttt{spawn}机制。
PyTorch在首次执行张量运算时初始化OpenMP线程池，
线程池内部持有互斥锁保护共享工作队列。
在\texttt{fork}模式下，子进程继承父进程的锁副本——
若分叉时刻恰好有线程持有该锁，
子进程将永久等待一个不会被释放的锁，形成死锁。
\texttt{spawn}模式通过启动全新解释器彻底规避了该问题：
子进程的OpenMP线程池从零初始化，不继承任何锁状态。
这是本系统选择\texttt{spawn}而非\texttt{fork}的根本原因。
