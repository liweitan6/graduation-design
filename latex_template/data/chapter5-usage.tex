当前实现中，各层之间的通信链路安排如下。
前端运行在Vite开发服务器上（默认端口5173），
默认通过Axios HTTP客户端访问Java Spring Boot后端。
分页查询、状态筛选、详情获取和统计概览指向Java后端（端口8080）；
向量计算、语义检索、RAG辅助诊断、Daikon候选边界条件分析、CEI分析和补盲建议由Python后端（端口5000）处理。
\texttt{/api/search}和\texttt{/api/ask}在Java后端里保留同名入口，
Controller通过Feign客户端调用Python路由，再把JSON结果透传回来。
数据导入和部分分析页面在前端配置了独立的Python服务Axios实例，
用来访问还没有经过Java代理封装的算法接口。

底层物理执行环境部署在WSL2中的Ubuntu 22.04子系统里。
这个子系统里安装了带Gcov探针的PyTorch编译版本，
以及独立的Python虚拟环境\texttt{venv\_pytorch}。
物理执行层通过\texttt{multiprocessing}模块的\texttt{spawn}启动方式创建独立的子进程，
在WSL环境下调用插桩后的PyTorch库执行测试脚本，
采集覆盖贡献数据。

\section{后端主要模块实现}

\subsection{Python FastAPI算法后端实现}

该服务以单文件形式组织（\texttt{data\_service.py}），
启动时由Uvicorn ASGI服务器监听5000端口。
服务初始化阶段建立Milvus连接、加载两个向量集合，
并加载预训练好的Sentence-BERT编码模型；MySQL连接则在处理具体请求时按需建立。

该后端注册了数据导入、语义检索、统计分析、候选边界条件分析、CEI清退和补盲建议等API路由。

下面以接口\texttt{/api/analysis/}\allowbreak\texttt{fault-boundary}
为例，说明请求体和响应体的JSON结构设计。

该接口的请求体只包含一个字段：

\texttt{\{"failed\_case\_uid": "<string>"\}}

其中\texttt{failed\_case\_uid}是目标崩溃用例的全局唯一标识。
接口内部执行以下步骤：
（1）从MySQL查询目标用例的完整元数据与向量主键；
（2）以向量主键在Milvus中提取结构特征向量；
（3）在Milvus中执行余弦相似度检索，召回20个最近邻样本；
（4）将近邻样本按执行状态分为健康组（最多10个）与崩溃组（最多10个）；
（5）对健康组与崩溃组执行参数展平与张量Diff对比；
（6）调用Daikon候选边界条件分析模块提取候选约束；
（7）根据候选约束构造满足约束与不满足约束的轻量样例，
并通过PyTorch算子级执行结果判断约束是否具有参考价值；
（8）将Diff结果、候选约束与验证报告共同注入LLM提示词，
生成面向开发者的辅助诊断报告。

响应体JSON包含六个顶层字段：

（1）\texttt{failed\_case}对象。
包含\texttt{uid}（崩溃用例标识符，字符串类型）
与\texttt{error}（原始错误文本，字符串类型）两个子字段。

（2）\texttt{successful\_case}对象。
包含\texttt{uid}（最近邻健康用例标识符）一个子字段。

（3）\texttt{diff}对象。
包含\texttt{successful\_values}与\texttt{failed\_values}两个子字段，
均为键值对字典，键为展平后的参数路径（如\texttt{layers.conv1.}%
\allowbreak\texttt{kernel\_size}），
值为该参数在健康用例和崩溃用例中的取值。
该字段供前端渲染参数对比视图。

（4）\texttt{daikon\_violations}数组。
每个元素为一条字符串形式的候选约束表达式，
例如\texttt{"layers.conv1.kernel\_size}%
\allowbreak\texttt{ + 2 * layers.conv1.padding[0]}%
\allowbreak\texttt{ >= layers.conv1.input\_size"}。
数组按变量数量降序排列，多变量约束排在前面。

（5）\texttt{daikon\_validation}数组。
每个元素对应一条候选约束，
包含\texttt{constraint}、\texttt{positive\_cases}、
\texttt{negative\_cases}与\texttt{is\_validated}四类信息。
其中\texttt{positive\_cases}记录满足约束式的生成样例，
\texttt{negative\_cases}记录违反约束式的生成样例；
每个样例进一步包含被变异的参数路径、变异后的参数值、
约束求值结果、实际执行状态以及报错摘要。
若所有正样例均能成功执行，且所有反样例均触发运行时错误，
则\texttt{is\_validated}字段置为真，
表示该约束在当前样本和轻量级验证样例上具有较高可信度。

（6）\texttt{boundary\_rule}字符串。
LLM基于上述上下文生成的辅助诊断报告内容。
提示词中包含\texttt{daikon\_validation}的执行验证结论，
使生成文本围绕候选边界条件展开，
减少将偶然相关关系误解释为源码级结论的情况。

数据导入接口\texttt{/api/ingest}采用POST方法，
请求体为TestCase对象的JSON数组。
每个TestCase对象包含以下字段：
\texttt{case\_uid}（字符串，全局唯一标识）、
\texttt{status}（字符串，执行状态枚举）、
\texttt{edge\_coverage}（整数，边覆盖数）、
\texttt{execution\_time}（浮点数，执行耗时秒）、
\texttt{error\_message}（字符串或空值，错误文本）、
\texttt{model\_structure}（嵌套字典，包含\texttt{layers}列表、
\texttt{connections}连接链、\texttt{depth}深度与\texttt{operators}算子列表四个键）。
响应体包含\texttt{success}（成功导入数量）、
\texttt{ids}（结构向量主键列表）与\texttt{message}（状态描述）三个字段。

这个接口体现了第4章所述的双路向量化设计：
服务根据\texttt{model\_structure}构造结构描述文本，
根据\texttt{error\_message}或成功占位文本构造日志描述文本，
再分别生成结构向量和日志向量。
结构向量写入\texttt{fuzzing\_structure}集合，
日志向量写入\texttt{fuzzing\_logs}集合。
当前Milvus集合只保存自增主键\texttt{id}和向量字段\texttt{embedding}，
\texttt{case\_uid}保存在MySQL表中；
跨库关联依靠\texttt{vector\_id}、
\texttt{parameters.structure\_vector\_id}和
\texttt{parameters.log\_vector\_id}。
检索时，Milvus先返回相似向量主键，
Python后端再用这些主键从MySQL反查完整元数据。
这一实现没有引入分布式事务，主要通过确定写入顺序和MySQL唯一键幂等写入来减少双库不一致的情况。

语义检索接口\texttt{/api/search}的请求体只包含一个字段
\texttt{query}（用户自然语言查询文本）。
接口内部先对查询文本作规则化的意图分类，
判断用户意图更偏向结构检索、日志检索还是复合检索，
再据此调整结构向量和日志向量的融合权重。
响应体包含\texttt{results}数组、\texttt{total}、\texttt{query}、
\texttt{search\_mode}与\texttt{query\_intent}字段。
\texttt{results}数组中每个元素包含用例的主要元数据、
相似度得分\texttt{similarity\_score}以及匹配来源\texttt{match\_source}；
\texttt{query\_intent}记录后端识别到的查询类型，
供前端决定结果说明和提示信息的展示方式。

\subsection{Java Spring Boot业务后端实现}

Java Spring Boot后端承担基础业务数据管理和前端请求组织功能，
监听8080端口，通过Spring Data JPA与MySQL交互。
这一后端的主要职责包括三项：
用例的分页查询和条件筛选、统计数据的聚合计算，
以及对Python后端中语义检索和RAG辅助诊断接口的代理转发。

用例分页查询接口\texttt{/api/cases}支持三个查询参数：
\texttt{page}（页码，默认0）、\texttt{size}（每页数量，默认10）、
\texttt{status}（可选的状态过滤条件）。
Repository层把这些参数转换成分页查询。
当\texttt{status}参数非空时，后端调用\texttt{findByStatus}按状态过滤；
否则调用\texttt{findAll}返回全量分页结果。
响应体采用Spring Data风格的分页封装，
包含\texttt{content}（当前页用例数组）、\texttt{totalElements}（总记录数）、
\texttt{totalPages}（总页数）与\texttt{number}（当前页码）四个字段。

统计接口组包含三条路由：
\texttt{/api/statistics/overview}返回用例总数、崩溃率和平均执行耗时；
\texttt{/api/statistics/status}返回各状态类别的用例计数；
\texttt{/api/statistics/errors}返回错误类型排行榜。
这三条路由中，状态分布和错误类型统计通过Repository中的原生SQL聚合查询完成，
总览统计算出用例总数、成功数、失败数、平均执行耗时和平均覆盖贡献数据，
再把结果封装为前端图表组件可以直接消费的JSON格式。

代理转发方式基于Spring Cloud OpenFeign组件。
Java后端在Controller层注册和Python后端同名的路由映射
（如\texttt{/api/search}、\texttt{/api/ask}）。
当请求到达时，Controller把请求体交给\texttt{PythonServiceClient}，
由Feign客户端转发到配置项\texttt{python-service.url}对应的Python服务，
再把Python后端的响应体原样返回给前端。
前端Axios超时时间设置为60秒，以适应LLM推理可能产生的较长延迟。

两个集合在服务启动时执行\texttt{collection.load()}操作，
把索引结构和向量数据加载到Milvus的查询节点内存中。
这一操作用来减少检索请求触发额外加载的开销，
实际延迟受数据规模、Milvus部署方式和硬件资源的影响。

跨库数据关联通过写入顺序和主键回填共同维护。
数据接入阶段按“先Milvus后MySQL”的写入顺序执行：
服务先把双路向量分别插入两个Milvus集合并拿到自动生成的主键，
再把结构向量主键写入\texttt{vector\_id}字段，
并把\texttt{structure\_vector\_id}和\texttt{log\_vector\_id}
写入\texttt{parameters} JSON字段。
MySQL侧依靠\texttt{case\_uid}唯一索引和
\texttt{ON DUPLICATE KEY UPDATE}语义在重复导入时更新已有记录。
CEI清退接口在执行模式下先按阈值筛出覆盖收益相对较低的记录，
再从MySQL中读取\texttt{vector\_id}、
\texttt{parameters.structure\_vector\_id}
和\texttt{parameters.log\_vector\_id}；
随后调用Milvus删除两个集合中的对应向量，
再执行\texttt{DELETE FROM test\_case\_metadata}移除结构化元数据。
由于MySQL和Milvus仍然不共享事务管理器，
本文把这一流程描述为级联清退和最终一致性维护，
而不是严格意义上的分布式事务保证。

\section{前端界面实现与功能展示}

本系统前端基于Vue 3组合式API（Composition API）和TypeScript构建，
使用Element Plus组件库组织表格、表单、抽屉和弹窗等控件。
整体界面采用深色背景（背景色\texttt{\#0f172a}）配合靖蓝主色调（\texttt{\#6366f1}），
主要用卡片、表格和弹窗来组织信息。
多数页面组件在\texttt{onMounted}生命周期钩子中触发初始数据拉取，
用户通过筛选、搜索、按钮点击或打开抽屉继续发起HTTP请求。
前端分别封装Java后端和Python后端的Axios实例：
用例管理和统计接口默认访问8080端口，
数据导入、分析和补盲建议等算法接口访问5000端口。

\subsection{系统总览仪表盘}

系统总览仪表盘（Dashboard）是用户进入系统后的首屏，
用来查看全局测试状态的聚合结果。
页面顶部排列四张统计卡片，展示累计用例数、通过率、崩溃总数和平均执行耗时。
卡片数据通过调用\texttt{/api/statistics/overview}接口获取，
接口返回的字段直接绑定到卡片的数值插槽上。

页面中部为测试状态分布环形图。
该图基于\texttt{vue-echarts}组件封装，
在\texttt{onMounted}钩子中调用\texttt{getStatisticsStatus()}函数拉取状态分布数据。
接口返回的数组格式为\texttt{[\{name: "Success", value: N\}, ...]}，
该数组赋值给ECharts配置对象的\texttt{series[0].data}属性。
ECharts内部基于Canvas 2D上下文渲染环形区域，
每个弧段的弧度由\texttt{value}字段占总和的比例决定。
环形图以通过与失败两种状态着色，支持鼠标悬停高亮。

\section{底层物理执行隔离实现}

本节描述物理层面的测试执行基础设施。
物理执行隔离层运行在WSL2 Ubuntu 22.04子系统中，
主要任务是：在带覆盖率探针的PyTorch编译版本上执行测试脚本，
采集C++层的分支命中数据，
通过进程级的执行隔离减少测试脚本崩溃对主服务的影响。

\subsection{Gcov/LCOV探针的编译挂载流程}

Gcov探针的工作原理基于编译期插桩和运行期计数两个阶段的协作。
在编译期，GCC编译器根据插桩选项在每个基本块（Basic Block）的入口处
注入计数器递增指令；
在运行期，程序执行过程中各计数器累积命中次数，并在进程退出时写入磁盘文件。

要对PyTorch底层C++源码启用Gcov插桩，
需要在WSL环境中修改编译选项。
具体方式是设置以下环境变量后重新触发CMake配置和编译：

\begin{verbatim}
export CXXFLAGS="-fprofile-arcs -ftest-coverage"
export LDFLAGS="-lgcov --coverage"
\end{verbatim}

其中\texttt{-fprofile-arcs}指令让编译器在每个分支跳转点插入弧计数器，
\texttt{-ftest-coverage}指令让编译器为每个编译单元生成\texttt{.gcno}文件。
链接阶段的\texttt{-lgcov}参数把Gcov运行时库静态链接到目标二进制中。

编译完成之后，每个\texttt{.cpp}源文件会对应产生一个\texttt{.gcno}文件。
这个文件记录源码的控制流图拓扑，
包括基本块编号、块间的有向边关系和源码行号映射。
\texttt{.gcno}文件在编译时一次性生成，之后不再变化。

当带探针的PyTorch库被测试脚本加载并执行之后，
运行时库会在进程正常退出（或收到\texttt{SIGTERM}信号）时，
把每个弧计数器的累积值写入和\texttt{.gcno}同目录的\texttt{.gcda}文件。
\texttt{.gcda}文件记录每条边在本次执行中被穿越的次数。
多次执行产生的\texttt{.gcda}数据会累积叠加。

覆盖率数据的提取通过LCOV命令行工具完成。
执行流程分为三步：

（1）基线采集。在任何测试执行之前，运行如下命令：
\begin{flushleft}
\small\ttfamily
lcov --capture --initial\\
\hspace*{1em}--directory /path/to/pytorch/build\\
\hspace*{1em}--output-file baseline.info
\end{flushleft}
生成零计数基线文件。

（2）增量采集。测试脚本执行完毕后，运行如下命令：
\begin{flushleft}
\small\ttfamily
lcov --capture\\
\hspace*{1em}--directory /path/to/pytorch/build\\
\hspace*{1em}--output-file test.info
\end{flushleft}
采集含有命中计数的覆盖率快照。

（3）差分计算。通过
\begin{flushleft}
\small\ttfamily
lcov --remove test.info '/usr/*'\\
\hspace*{1em}--output-file filtered.info
\end{flushleft}
过滤系统头文件贡献后，
再利用如下命令：
\begin{flushleft}
\small\ttfamily
genhtml filtered.info\\
\hspace*{1em}--output-directory coverage\_html
\end{flushleft}
生成可视化HTML报告。

LCOV报告中的覆盖统计为本系统的\texttt{edge\_coverage}字段提供物理来源。
在实现中，Python侧解析\texttt{.info}文件后提取可以用于度量覆盖贡献的数据，
再以整数形式回写到MySQL对应的用例记录中。
这一字段在CEI模型中对应$E_{\text{edge}}$，
用来表示单个测试用例贡献的覆盖规模。

\subsection{基于multiprocessing.spawn的隔离执行模块}

测试脚本的执行环境主要会遇到两类异常情况：
一是脚本可能触发底层C++算子的段错误（Segmentation Fault），
导致整个进程地址空间被操作系统回收；
二是PyTorch内部依赖的OpenMP线程池在\texttt{fork}模式下
可能继承父进程的锁状态，形成无法恢复的死锁。
如果测试脚本和主Web服务共享进程空间，上述情况可能导致后端服务异常退出。

为了减少上述影响，本系统采用\texttt{multiprocessing}模块的\texttt{spawn}启动方式。
和默认的\texttt{fork}模式不同，\texttt{spawn}不复制父进程的内存页表，
而是启动一个全新的Python解释器进程，
从零开始导入目标模块并执行指定的函数。
这种方式让子进程使用独立的地址空间、线程池和GPU上下文。

启动方式通过调用
\texttt{multiprocessing.set\_start\_method}%
\allowbreak\texttt{('spawn', force=True)}全局设定。
\texttt{force=True}参数会覆盖可能已被第三方库设置的启动模式。
之后所有通过\texttt{multiprocessing.Process}创建的子进程
都以\texttt{spawn}方式启动。

每个测试脚本的执行流程如下。
主进程构造\texttt{Process}对象，把执行函数和参数序列化后传入：

\begin{verbatim}
p = multiprocessing.Process(
    target=_execute_wrapper,
    args=(case_uid, input_shape_str, operators_str)
)
p.start()
p.join(timeout=10)
\end{verbatim}

\texttt{p.join(timeout=10)}让主进程阻塞等待子进程完成，
最长等待10秒。如果子进程在超时时限内没有退出，
主进程会调用\texttt{p.terminate()}发送\texttt{SIGTERM}信号终止子进程，
再调用\texttt{p.join()}等待资源回收完成。
当前物理执行脚本已经实现了这一超时控制，
用来防止死循环或底层算子长时间阻塞导致的资源泄漏。

子进程退出之后，主进程通过\texttt{p.exitcode}属性获取退出状态码。
退出状态码的语义遵循POSIX标准：
值为0表示正常退出；
正整数表示Python层未捕获的异常导致的退出（例如值1为通用错误）；
负整数表示子进程被操作系统信号终止，绝对值就是信号编号。

当底层C++算子因为内存越界触发段错误时，
Linux内核会向子进程发送Signal 11（\texttt{SIGSEGV}）。
由于子进程没有注册这一信号的处理函数，
默认行为是立即终止进程并产生核心转储（如果启用了核心转储）。
这时主进程观察到\texttt{p.exitcode == -11}。

主进程可以根据退出状态码作分类处理。
当前批量物理执行脚本主要完成隔离执行、超时回收和探针数据注入；
如果需要把执行状态回写到数据服务，可以采用以下分类逻辑：

\begin{verbatim}
if p.exitcode == 0:
    status = "SUCCESS"
elif p.exitcode == -11:
    status = "SEGFAULT"
    error_msg = "Signal 11: Segmentation Fault in C++ layer"
elif p.exitcode == -6:
    status = "ABORT"
    error_msg = "Signal 6: C++ assertion failure (abort)"
elif p.is_alive():
    p.terminate()
    p.join()
    status = "TIMEOUT"
else:
    status = "CRASH"
    error_msg = f"Exit code: {p.exitcode}"
\end{verbatim}

Signal 11（段错误）和Signal 6（\texttt{SIGABRT}，断言失败）
是PyTorch底层算子在非法参数组合下可能触发的两类信号。
系统可以根据\texttt{p.exitcode}记录执行状态，
相关结果用于后续的统计、故障辅助定位、候选边界条件分析和RAG辅助诊断。

这种执行隔离方式的主要作用是：
不论子进程以哪种方式终止——正常退出、异常退出或被信号杀死——
主进程通常不会跟着子进程一起退出。
FastAPI的ASGI事件循环仍然可以继续响应其他HTTP请求，
从而减少单个测试脚本的C++层崩溃对服务进程的影响。

OpenMP线程死锁的处理同样依赖\texttt{spawn}启动方式。
PyTorch在首次执行张量运算时会初始化OpenMP线程池，
线程池内部持有互斥锁来保护共享的工作队列。
在\texttt{fork}模式下，子进程继承父进程的锁副本，
如果分叉的时刻恰好有线程持有这个锁，
子进程就会永久等待一个不会被释放的锁，从而形成死锁。
\texttt{spawn}模式通过启动全新解释器避免继承这一锁状态：
子进程的OpenMP线程池从零初始化，不继承任何锁状态。
这是本系统选择\texttt{spawn}而不是\texttt{fork}的主要原因。
