## 本科毕设题目

### 题目内涵的简要解读

**题目：面向深度学习模糊测试的测试用例智能管理与效能分析系统**

做一个深度学习框架模糊测试产生的信息的管理和分析系统（信息包括测试用例和日志）

- 导入测试用例以及日志信息，测试用例包含两部分，计算图结构和初始参数，日志中包含的信息见另一个文档。
- 对测试用例进行分析
  - 比如对数据库中目前的测试用例进行聚类，或找某种度量来表征计算图之间差异度，看库里的生成器生成数据的多样性如何
  - 比如根据日志中的覆盖边数相关的信息，分析单个测试用例的覆盖效果，如果可能的话，尝试分析出什么样的测试用例的覆盖效果更好
  - 这里可能涉及到一些前后端的开发，可视化效果可能要做一些。
- 能够自然语言去查包含某些特征（包含哪些算子、包含什么样的图结构、是否失效）的测试用例，从而帮助科研人员快速找出一些测试用例。这里可能涉及到 LLM、RAG 之类的技术。

> 事实上，只要你做的东西能对应上题目的关键词，就可以按照自己的想法去做，不必限于上面说的这几点，也不必上面说了啥就一定做。

### 背景与目标

**背景：** 在深度学习框架（如 PyTorch）的模糊测试研究中，测试工具（Fuzzer）会生成海量的测试用例（包含计算图结构和参数）及相应的运行日志。

**痛点：**

1. 数据量大，测试用例结构（计算图）与运行日志（错误信息）往往割裂，缺乏有效的深层映射和回溯手段。
2. 缺乏对生成器“多样性”的直观评估（是不是总是生成一样的图？），以及对“覆盖率缺口”的精准识别（还有哪些算子组合没测到？）。
3. 缺乏对“测试效能”的量化分析（什么样的图覆盖率高且结构精简？）。
4. 测试结果往往停留在“数据展示”层面，缺乏能够直接指导新用例生成的闭环优化建议。

**目标：** 开发一个集**数据管理、可视化分析、智能检索**于一体的 Web 系统，辅助科研人员评估生成算法优劣并快速定位缺陷。

### 工具可能包含的核心功能

#### 数据解析与结构化存储

**结构化日志解析与存储**

- **功能描述：** 彻底改变将日志作为“黑盒”文本存储的做法，对其进行深度的结构化提取。
- **功能点：**
  - **多维信息提取：** 利用正则匹配和抽象语法树，提取计算图中的算子对、张量维度（如 `[32x512]`）、框架源码出错位置、以及显存消耗（如 `512 MiB`）等深层语义信息。
  - **独立日志管理：** 建立专门的 `test_case_logs` 表记录完整日志，并在 UI 层支持带语法高亮的聚合对比。
  - **存储策略：**
    - 元数据及结构化特征（ID、状态、算子列表、覆盖率、效能指标） $\rightarrow$ **关系型数据库**。
    - 文本及向量映射（图描述、源码、日志 Embedding） $\rightarrow$ **向量数据库**。

#### 测试辅助与效能分析（体现对框架测试的作用）

- **功能描述：** 从简单的“数据展示”升级为“主动发现分析”，指引后续的测试方向。
- **功能点：**
  - **覆盖率缺口分析 (Coverage Gap Analysis)：**
    - 构建“算子对”覆盖热力图（Heatmap），直观呈现哪些算子组合已经被充分测试，明确指出**未覆盖的算子组合盲区**，指导测试人员“下一步该测什么”。
  - **测试效能指数 (Coverage Efficiency Index, CEI)：**
    - 结合图结构复杂度，定义效能指标 `CEI = 覆盖边数 / 算子复杂度分布`。
    - 通过散点图定位出“结构精简但覆盖率高”的高价值用例，作为未来回归测试的最佳种子。
  - **自动故障定位与深层错误关联：**
    - 将解析出的错误日志与测试用例的 DAG 结构关联。例如：Shape Mismatch 映射到维度变换算子（Linear, Conv2d）；OOM 映射到特定大算子。
    - 错误模式聚类分析：生成类似“`Conv2d (大kernel) + BatchNorm2d 组合在特定维度下导致 OOM 概率 80%`”的因果级系统分析报告。

#### 基于 RAG 的智能检索助手

- **功能描述**： 允许用户使用自然语言查询测试用例库。
- **功能点：**
  - **RAG（检索增强生成）流程实现：**
    1. **用户提问：** 帮我找一个包含 Conv2d 算子且导致 Segmentation Fault 的测试用例。
    2. **语义检索：** 系统将问题转化为向量，在向量数据库中匹配最相似的测试用例记录。
    3. **LLM 总结：** 将检索到的 top-3 用例的摘要喂给 LLM，让 LLM 生成最终回答并列出具体的用例。
  - **应用场景：** 快速筛选特定类型的测试用例样本，无需写复杂的 SQL。

#### 日志驱动的 AI 测试用例变异建议

- **功能描述**： 基于历史关联数据的因果链推理，利用 LLM 为测试人员生成新的测试用例变异方向。
- **功能点：**
  - **AI 建议生成：** 将历史的“错误模式”和“覆盖缺口”作为 Context 投喂给大模型。
  - **变异输出：** 输出带有针对性的建议（如：“建议针对过往高频触发 Shape Mismatch 的 Conv2d->Linear 组合插入 Flatten 层生成变体”）。
  - **对比直接 AI 优化的优势：**
    - **可解释性与可追溯性：** 建议紧密依附于历史测试用例和真实 Log 证据，并非黑盒猜测。
    - **知识积累机制：** 数据和错误模式积累越多，系统的 AI 建议越准确，实现人机协同的测试优化闭环。

### 工具的意义

1. **多样性评估工具：** 可视化的结果能够一眼看出 Fuzzer 生成的图是不是重复或千篇一律。如果点都聚在一坨，你就知道需要改进变异策略了。
2. **高质量种子筛选：** 通过效能分析，可以直接导出那些覆盖率高或覆盖效率高的测试用例，作为下一轮模糊测试的**初始种子**，从而加速后续的测试过程。
3. **Case Study 素材库：** 当需要举例说明“某类结构容易导致Bug”时，直接用自然语言功能搜一下，马上就能得到代码和报错信息，省去翻日志的痛苦。

### 一些建议

- **关于计算图解析：** 可以先不需要写复杂的 Parser 去理解图的拓扑逻辑。直接把图结构文件当作**文本**，丢给一些模型（不一定是大模型）变成向量即可。
- **关于 LLM：** 毕设重点在于应用，不要去训练或微调大模型。直接调用 API 做 RAG 即可。
- **关于数据：** 先用少量的日志把系统跑通，再考虑导入全量数据。

### 关于开题报告

#### AI框架的重要性

- **故事线：** 深度学习框架（PyTorch, TensorFlow）是现代 AI 系统的基础设施。如果框架本身有 Bug（例如计算梯度错了一点点），那么上层的自动驾驶、医疗 AI 都会产生致命后果。
- **引出测试：** 为了保障安全，必须进行大规模的**模糊测试（Fuzzing）**。

#### AI框架测试的实验痛点

- **故事线：** 现在的自动化工具每天能产生几万个测试用例。但是，这些数据大多是杂乱无章的 `.py` 文件和 `.log` 文本。
- **矛盾点：** 研究人员想复现一个 Bug，或者想看测试覆盖全不全，就像在“大海捞针”。

#### 你要做的事情：智能管理与分析测试用例


**回答开题 3 个核心拷问：**

1. **主线，本项目对 AI 框架测试的实际作用是什么？**
   - 不仅仅是存数据画饼图。系统能通过**覆盖率缺口热力图**直接告诉测试团队“目前框架的哪些算子拓扑组合还存在盲区”；通过计算**测试效能 (CEI)** 筛选出性价比极高的精简测试种子；并通过**错误模式聚类**发现导致框架 OOM 或 Segmentation Fault 的系统性规律，真正反哺测试策略。

2. **如何充分利用测试用例和 Log 文本的作用？**
   - 彻底打破以往结构与日志割裂的局面。采用**结构化日志解析**（正则提取维度异常、代码行位置、显存占用），并实施**自动故障定位**，将日志中的 Error 追溯映射到测试用例计算图（DAG）的特定“嫌疑算子”上，形成由因到果的闭环关联链条。

3. **关于测试用例 AI 优化？用本项目进行 AI 建议与直接用黑盒 AI 优化相比好处在哪？**
   - 直接用 AI 生成测试用例属于“黑盒投喂”，存在不可预测性且难以定位为何生成；而本系统是**基于海量历史执行数据的分析和错误沉淀**作为 RAG 上下文去给大模型下达 Prompt 指导变异。
   - **核心优势在于：可解释性、可追溯性和知识累积**。给出的每一个变异优化建议（如：针对高危组合 Conv2d+BatchNorm 推荐生成什么样的数据），都能精准追溯关联到特定的几宗历史崩溃案卷证据链上，既能给生成器算法提供调优灵感，也能允许工程师辅助判断和人工介入。
