# 毕业设计周进展情况记录（第 10-16 周计划与实施）

**课题名称**：面向深度学习模糊测试的测试用例智能管理与效能分析系统  
*(注：第 10-16 周为主体研发结束后的冲刺、论文撰写与答辩准备阶段)*

---

## 第 10 周进展情况记录

### 本周完成/计划情况：
1. **扩大物理验证规模**：在前期 200 例 A/B 物理测速的基础上，利用 `real_cei_executor.py` 发起更大规模的 1000 例深度物理跑测。进一步验证系统在长时间高负载 Fuzzing 环境下的防彻底死锁能力与 CEI 效能截断的稳定性。
2. **Daikon 双空间故障边界定界引擎稳定性回归**：在大规模数据集上对 `boundary_analyzer.py` 的"健康组/崩溃组"召回与参数展平流程做稳定性回归，校核"仅当成功样本全票通过且失败样本全票否决"时方可提取数学红线的硬约束逻辑，避免大样本量下出现伪边界公式。
3. **前端渲染性能重构**：针对 ECharts 渲染万级 PCA 散点出现的掉帧与卡顿问题，引入 ECharts GL (WebGL) 底层渲染策略，实现十万级测试用例降维分布的丝滑缩放，覆盖 `Diversity.vue` 与 `Coverage.vue` 双页面。
4. **论文核心章节动笔**：基于上周跑出的详实数据与图表，正式撰写毕业论文的《系统架构与核心算法设计》以及《实验论证与效能分析》两章核心内容，重点论述 Daikon 双空间定界与 CEI 物理评估两条主线。

### 存在的问题：
在扩大测试规模后，MySQL 与 Milvus 数据库的双写同步压力增大，偶尔出现极个别短时数据状态不一致的现象；同时 Daikon 引擎在 1000 例规模下扫描耗时显著上升。

### 下周计划：
在代码层面上增加消息队列或补偿轮询机制来保证数据最终一致性；对 Daikon 模板扫描做剪枝与并行化优化；开启毕业论文前三章（绪论、相关工作、需求分析）的撰写。

## 第 11 周进展情况记录

### 本周完成/计划情况：
1. **DFS 拓扑故障溯源前后端联调**：完成 `fault_locator.py` 逆向邻接矩阵 DFS 引擎与前端 `Correlations.vue` 的全链路对接，将"嫌疑算子集合 + 置信度"结果实时高亮渲染至故障溯源面板，与 Daikon 双空间边界对比窗形成"硬推断 + 软解释"的双轨展示。
2. **算子缺口靶向补盲闭环（LLM Auto-Generate）**：联调 `/api/auto-generate-from-gaps` 链路，由 `Coverage.vue` 热力盲区一键触发——后端基于算子组合差集封装框架约束，驱动 DeepSeek 生成定制 PyTorch 测试代码，实现"特征抽取 → 缺陷盲区反馈 → LLM 定向用例生成"的三位一体智能化循环。
3. **大语言模型输出合规化**：针对 DeepSeek 自动生成的补盲 PyTorch 代码可能存在的语法或过期 API 风险，在 LLM 自动交互环路中增加"语法树抽象提取验证（AST Validation）"机制，确保送入后端执行的生成代码基本合规。
4. **CEI 双库事务级级联清退**：在 `Efficiency.vue` 智能清退控制台与 `/api/analysis/efficiency/prune` 后端联通基础上，补齐 MySQL 元数据与 Milvus 向量记录的事务级一致清理，杜绝清退后向量库脏读残留。
5. **全链路端到端联调测试**：打通 Vue3 交互面板指令 → Java CRUD 数据交互 → Python 5000 端口（PCA/CEI/Daikon/LLM 运算）→ 触发 WSL 底层 Gcov C++ 物理流的完整体系，消除各端口间的跨域（CORS）与数据包装隐患。

### 存在的问题：
LLM 自动生成的补盲代码即便通过 AST 静态校验，仍偶发出现运行时维度不匹配错误，缺少端到端的执行级合规闭环。

### 下周计划：
开发 Daikon 红线驱动的闭环自动化变异器原型，将"硬约束公式"反向注入用例生成器；同步推进论文中后段章节的拼装。

## 第 12 周进展情况记录

### 本周完成/计划情况：
1. **闭环自动化变异器（Auto-Loop）原型验证**：基于 Daikon 引擎输出的"安全边界公式"，开发轻量 Mutator，对崩溃用例自动生成满足红线约束的修复变体并送回 `real_cei_executor.py` 重测，初步形成"崩溃 → 红线 → 修复变异 → 回测"真闭环 Fuzzing 链路。
2. **Daikon 模板池扩充**：在 `boundary_analyzer.py` 中新增内存分配（Strides/Contiguous）相关的底层优化扫描模板，覆盖范围由原先的"单变量阈值、奇偶性、双变量等式整除、三变量空间坍缩"进一步扩展至 stride-shape 一致性与 channel 整除等约束，提升红线提取的可解释性。
3. **论文初稿拼装**：完成论文《系统实现与微服务联动机制》、《总结与展望》章节，并将前端 UI 截图、RAG 问答链路图示、Daikon 红线公式表、V3 版本物理 A/B Test 校准图全量排版插入至论文正文中。

### 存在的问题：
论文的论述部分结构过于口语化或偏向工程实战，缺乏学术化的包装与严密的理论推演公式陈述。

### 下周计划：
完成论文初稿的自校与外文文献翻译；针对系统整体的代码仓库进行脱敏、瘦身与代码 Freeze；准备一份完整的初稿与项目白皮书提交给指导老师进行第一轮审阅。

## 第 15 周进展情况记录

### 本周完成/计划情况：
1. **学校查重（知网）与降重定稿**：提交毕设终稿至学校指定平台进行查重检验。针对个别在综述部分出现的常规重复句法进行学术化的缩减与同义词更迭，顺利将查重率压降至 5% 的优秀区间以下。
2. **毕设答辩材料精加工（PPT 制作）**：围绕"为什么做（痛点）→ 核心技术怎么做（Daikon 双空间定界 / CEI 模型 / 物理隔离沙箱 / LLM 闭环补盲）→ 效果如何（A/B Test 覆盖率与机时对冲图、Daikon 红线公式表）"为主轴，制作具备极强逻辑递进关系的高质量汇报演示 PPT。
3. **录制系统操作展示全流程回放视频**：由于答辩现场网络环境不可控（可能无法调用 DeepSeek API 或启动沉重的 WSL 编译环境），精心录制了一段长约 4 分钟的系统全链路操作演练实况录屏（覆盖故障溯源高亮、Daikon 边界对比、CEI 一键清退、PCA 多样性聚类、LLM 补盲生成等核心面板），作为答辩兜底。
4. **全真模拟预答辩演示**：与同组同学开展多次互相听讲预演，精准把控演讲用时。打磨"受到挑战性提问"时的临场反应，重点准备了针对"CEI 分数判定科学性"、"Daikon 双空间红线提取的统计严谨性"、"大模型代码执行防毒方案"、"DFS 故障溯源置信度分配依据"等预设答辩防御点。

### 存在的问题：
答辩汇报规定的时限通常为 8~10 分钟，目前准备的内容从前端页面讲解到底层 C++ 探针逻辑跨度极大，极易造成超时。

### 下周计划：
舍去部分非核心业务的增删改查前端介绍，将答辩着力点 80% 压实在"物理效能验证突破"和"大语言模型自闭环"这两个答辩评委最关注的技术深水区，进行多轮掐钟预演；参加校级正式答辩并完成最终归档。
