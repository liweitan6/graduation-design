## 一、项目概述

**项目名称**：面向深度学习模糊测试的测试用例智能管理与效能分析系统

**课题背景**：针对深度学习框架（如 PyTorch）的底层 API 和算子组合，进行基于模糊测试（Fuzzing）的安全性验证、故障定界以及边界红线推断。系统摒弃了“将报错日志直接丢给大模型盲猜”的偷懒做法，而是建立了一条**“双空间算法硬推断 + 大模型软解释”**的双轨排障流水线。系统融合了经典的软件工程分析理论（如差分调试、Daikon 不变量推断）和前沿的大模型（LLM）能力，核心学术价值在于“优雅、严谨地在物理和数学层面证明框架崩溃的红线”。

## 二、系统架构

系统采用**前后端分离 + 混合后端架构**，由以下层级构成：

### 技术栈

| 层级 | 技术 | 职责 |
| --- | --- | --- |
| 前端 | Vue 3 + TypeScript + Element Plus | 极客风/玻璃拟态 UI，交互式大盘、故障溯源与边界对比实验窗 |
| Java 后端 | Spring Boot + MyBatis | 结构化业务数据（基础 CRUD、路由转发与兼容性代理） |
| Python 后端 | FastAPI（5000端口） | 核心算法侧与 AI 任务：Daikon 推断、向量化、日志解析、LLM 调用 |
| 双库存储层 | MySQL + Milvus 2.3.x | MySQL 存元数据（状态/覆盖率/日志）；Milvus 存算子拓扑的高维语义向量 |
| 物理采集层 | GNU Gcov / LCOV 工具链 | 底层 C++ 真实代码行覆盖率追踪与插桩验证 |

### 关键依赖
- **Embedding 模型**：`all-MiniLM-L6-v2`（384 维），提取算子拓扑结构特征
- **LLM**：DeepSeek-V3，大模型作为“翻译官”解释底层公式
- **向量索引**：Milvus `IVF_FLAT`，余弦相似度（COSINE）检索相似性克隆体
- **启动方式**：`e:\BUAA\graduation-design\start-all.ps1`

### 服务端口
- 前端：`http://localhost:5173`
- Java API：`http://localhost:8080`
- Python API：`http://localhost:5000`

## 四、核心研究模块与算法引擎

### 4.1 自动化日志解析与双路向量化数据流水线
- **LogParser**：基于分层正则匹配机制，过滤深层冗余调用帧，针对 8 种标准化框架错误定向提取错误大类、张量维度、显存状态与报错源码行，转化为结构化 JSON。
- **双路向量化表征**：基于预训练的孪生网络架构（Sentence-BERT），将离散的长文本报错堆栈与多维嵌套的计算图拓扑序列，统一降维投射为 384 维稠密浮点向量矩阵，打破跨语言调用的文本检索困境。

### 4.2 异构混合协同存储与削峰调度机制
- **异构双库解耦**：采用结构化元数据（MySQL）与稠密特征向量（Milvus）的隔离协同设计，利用全局唯一标识符实现跨系统的强关联。
- **异步调度与生命周期管理**：设计了基于内存缓冲与异步协程的流量削峰机制，保障高并发双写一致性。支持基于 CEI 效能阈值的跨库级联清理，防止数据脏读。

### 4.3 故障溯源与 Daikon 双空间故障边界定界（核心亮点）
- **DFS 拓扑故障溯源 (Fault Locator)**：当崩溃发生时，引擎依据计算图逆向邻接矩阵执行深度优先遍历。结合张量约束逐层溯源，精确计算“嫌疑算子集合”及置信度并在前端高亮。
- **双空间 Daikon 边界定界引擎**（位于 `boundary_analyzer.py`）：
  - **对照组召回**：检索召回 10 个拓扑相同但成功的“健康组”与 10 个“崩溃组”。
  - **参数展平与物理探针**：将嵌套的张量参数展平为一维数值键值对。
  - **绝对数学规律探测**：通过扫描测试空间坍缩等海量算子规则（如 `input_size + 2*padding >= kernel_size`）。**仅当某公式在成功样本全票通过且失败样本全票否决时**，提取为解释框架崩溃的“绝对数学红线”。

### 4.4 测试集多样性度量与 CEI 效能物理评估
- **PCA 降维监测**：引入主成分分析，将 384 维特征投射至二维平面。前端利用 WebGL 大规模散点图直观呈现同质化聚类簇，揭示生成器引擎的局部最优变异死锁。
- **CEI 效能指数计算**：利用 `CEI = EdgeCoverage / sum(OperatorWeight * Count)`，综合衡量测试脚本的分支探索贡献与计算成本，精准识别低收益用例。
- **底层物理防线验证**：在 WSL 下成功编译携带 Gcov/LCOV 探针的底层 PyTorch。开发了基于 `multiprocessing.spawn` 的隔离沙盒以防止死锁。

### 4.5 基于检索增强生成 (RAG) 的智能化闭环
- **环境感知辅助诊断**：将检索到的相似故障上下文及 Daikon 数学红线注入提示词，大模型针对严格项目环境输出具备自然语言解读的排障分析报告。
- **算子缺口靶向补盲**：提取从未使用的边缘算子组合对。后台封装框架约束指令，驱动 LLM 生成定制的 PyTorch 测试代码，实现“特征抽取 -> 缺陷盲区反馈 -> LLM 定向用例生成”的三位一体智能化循环。

## 七、后续待接手方向（Action Items）

1. **闭环自动化 (Auto-Loop Test Generation)**：
   依据大模型输出的“修复建议”或“安全边界公式”，开发变异器（Mutator）自动生成**修复后**的新测试用例送回框架重测，形成真正的 Fuzzing 闭环。

   在 `boundary_analyzer.py` 中增加更多关于内存分配（Strides）等底层优化相关的扫描模板（目前已涵盖单变量阈值、奇偶性、双变量等式整除、三变量空间坍缩）。

## 九、开发约定

- **学术严谨性**：优先考虑在论文中如何用软件工程理论自圆其说，切忌一切黑盒化。
- **UI 风格**：极客风/玻璃拟态（Glassmorphism），深色主题，主色调 `#6366f1`（靛蓝），背景色 `#0f172a`。
- **通信链路**：
  - 前端 -> Python 后端（5000 端口）：直接调用，用于所有分析类与 AI 密集任务。
  - 前端 -> Java 后端（8080 端口）：用于基础用例 CRUD 与简单统计。
- **底层探针调用**：若需真机收集，需进入 WSL 及对应 `venv_pytorch` Python 环境执行编译和插桩。
