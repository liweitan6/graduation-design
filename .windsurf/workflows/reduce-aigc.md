---
description: 对中文学术 / 技术文档做语言平实化（降 AIGC）改写
---

本 workflow 用于对中文文档（LaTeX、Markdown、纯文本等）做轻量、局部的语言改写，目的是降低 AIGC 检测命中率、去掉"AI 包装感"，**不改动任何技术内容、数据、公式、图表、引用、字段名**。

适用场景：学位论文、技术报告、API 文档说明段落、博客技术文章等。

---

## 0. 启动前确认（必做）

每次执行前先与用户对齐以下三项，写入工作备忘后再开始改写：

1. **目标范围**：哪些目录、哪些扩展名（`.tex` / `.md` / `.txt` / `.rst`）需要处理；哪些（如英文段落、附录、外部库代码、自动生成内容）需要排除。
2. **项目专有术语表**：本项目希望统一使用的中文术语和英文缩写写法。如果用户没有现成列表，先扫描文档抽取候选词，再请用户确认后填入第 3 节。
3. **项目专有禁用词**：本项目过去被标记为"包装"或"夸大"的具体词条，追加到第 2.1 节的基础禁用列表里。

> 没有这三项确认就进入改写，会出现术语漂移、误删项目特定表达的风险。

---

## 1. 改写范围

**处理：** 中文叙述性段落、摘要、引言、相关工作、方法描述、实验解读、总结展望等。

**严禁改动：**

- 英文段落、英文标题、英文关键词（除非用户显式要求）
- 表格内的数据行（`\begin{tabular}` / Markdown 表格主体）
- 公式（`equation` / `align` / `$...$` / `\[...\]` / `$$...$$`）
- 算法伪代码（`algorithm` / `algorithmic` 环境、代码块 ` ``` `）
- 引用、标签、交叉引用键（`\cite` / `\label` / `\ref` / `\eqref` / `[@key]`）
- 接口路径、命令行、文件名、类名、函数名、字段名、配置键、URL
- 任何数值（百分比、耗时、行数、内存、维度、版本号等）
- 章节标题、节标题、图表标题、目录条目
- 排版命令本身（`\texttt`、`\textbf`、`\allowbreak` 等）
- 参考文献条目（`\bibitem` / `.bib` 文件）

## 2. 三类禁用表达

### 2.1 包装 / 营销型词汇（基础类，必删或改写）

**通用基础禁用类：**

- 智能化、一站式、全方位、一体化（作修饰语时）
- 革命性、颠覆性、首创、首次提出、突破性、里程碑式
- 充分证明、完全解决、彻底实现、显著优于（无具体数据支撑时）
- 工业级、企业级、生产级（无对应实验环境支撑时）
- 闭环、赋能、抓手、生态、范式（互联网营销词）

**项目扩充类：** 由用户在第 0 节启动确认时追加，写入下方代码块：

```
<在此追加项目专有禁用词，每行一个或顿号分隔>
```

### 2.2 高级修辞 / 华丽句式（应改写为平实表达）

- 排比 / 三段式平行结构（"既……又……更……""不仅……而且……还……"）
- 修辞性形容词堆叠（"显著""卓越""强大""高效""鲁棒""核心""关键"）
- 夸张定语（"本文创新性地……""本文首次提出……"）
- 连续 4 个以上名词的技术清单堆叠 → 用上位词概括或保留 2~3 项关键名词后用"等"收束
- 长定语链（"基于……的……的……机制"超过 25 字）

### 2.3 书面化连词（建议替换为口语化连词）

| 原表达 | 改写为 |
|---|---|
| 于 / 在……过程中 | 在 |
| 与 / 及 / 以及（句中堆叠时） | 和 |
| 为（表判断） | 是 |
| 将……进行 / 对……进行 | 把……做 / 对……做 |
| 通过……方式实现 | 用……做 / 通过……完成 |
| 旨在 / 致力于 | 目的是 / 用于 |
| 从而……进而…… | 这样…… |
| 鉴于……因此…… | 因为……所以…… |

## 4. 操作流程

### 步骤 1：定位目标文件

用 `find_by_name` 或 `list_dir` 列出目标目录下匹配扩展名的文件，再排除第 0 节确认的排除项。

### 步骤 2：逐文件 read + 定位需改写段落

每次只处理一个文件，先用 `read_file` 通读，按以下三类标记需改写的句子：

1. **名词堆叠**：连续 4 个以上名词 → 用上位词概括，或保留 2~3 项关键名词
2. **书面化 / 华丽**：含书面化连词、修辞形容词或排比结构 → 按 2.2 / 2.3 替换
3. **禁用词**：命中第 2.1 节或项目扩充禁用列表 → 删除或改为统一术语

记录改写候选时，保留原句行号和上下文，便于后续 `multi_edit` 定位。

### 步骤 3：用 `multi_edit` 小批量改写

**关键经验：** 单次 `multi_edit` 改写过多会触发 `deadline_exceeded`。

- 每次 `multi_edit` 控制在 **8~12 条 edit** 以内
- 每条 `old_string` 保留足够上下文以保证在文件内唯一匹配
- 严禁改动 `old_string` 中的公式、引用、命令、字段名、数值
- 禁用 `replace_all`，逐句精确替换
- 改完一个文件再进下一个，避免文件间互相干扰

### 步骤 4：编译 / 渲染验证

按文件类型执行项目自身的验证命令：

- **LaTeX**：项目使用的 `xelatex` / `latexmk` / `make` 命令，看 error 和 Overfull 警告
- **Markdown**：项目使用的预览器或 `pandoc` 渲染，看是否破坏了表格 / 代码块
- **通用**：`git diff --stat` 检查行数变化是否合理（一般每改一句仅 ±0~2 行）

> 具体命令在第 0 节启动确认时由用户提供，本 workflow 不预设。

### 步骤 5：自检禁用词

PowerShell 示例（按项目情况替换 `<目标目录>` 和扩展名）：

```powershell
$forbidden = @(
    # 通用基础类
    '智能化','一站式','革命性','首创','突破性','充分证明','完全解决','工业级','闭环','赋能',
    # 在此追加项目扩充类
    '<项目禁用词1>','<项目禁用词2>'
)
foreach ($w in $forbidden) {
    $hits = Select-String -Path '<目标目录>\*.<扩展名>' -Pattern $w -SimpleMatch
    if ($hits) { Write-Host "[HIT] $w"; $hits }
}
```

类 Unix 环境用 `grep_search` 工具按词逐项检查也可。期望输出为空。

### 步骤 6：差异回看

执行 `git diff --stat` 看改动文件清单，再 `git diff <file>` 抽样回看，确认：

- 行号偏移在合理范围内
- 没有意外修改公式、引用、字段名、数值
- 章节标题、节标题、目录条目未变动
- 表格、代码块、伪代码内容完整

## 5. 改写示例（通用风格）

### 5.1 名词堆叠 → 概括

**改前：** "系统从用户接口、配置中心、日志采集、监控告警、调度引擎和数据仓库等多个组件获取信息"

**改后：** "系统从接口层、配置中心、日志和监控等组件获取信息"

### 5.2 书面化 → 平实

**改前：** "本文将检索结果与推断模块相结合，旨在为后续分析提供更全面的支持"

**改后：** "本文把检索结果和推断模块结合起来，为后续分析提供支持"

### 5.3 华丽修饰 → 中性

**改前：** "该方法相较于传统方法具有更强的鲁棒性，能够显著降低误报率，充分证明其优越性"

**改后：** "该方法对噪声样本不敏感，可以降低误报率，具体数据见 `<引用>`"

### 5.4 排比 → 拆分

**改前：** "系统既保证了高效率，又兼顾了易用性，更实现了可扩展性"

**改后：** "系统在效率上做了优化，同时保留了较好的易用性和可扩展性"

### 5.5 长定语 → 拆句

**改前：** "本文提出了一种基于多源异构数据融合的面向领域知识图谱的自动化构建方法"

**改后：** "本文提出了一种领域知识图谱构建方法，输入是多源异构数据，构建过程自动完成"

---

## 6. 完成判定

- 所有指定文件已逐段过完一遍
- 项目自身的编译 / 渲染流程通过，无新增 error
- 步骤 5 的禁用词检查输出为空
- 章节结构、公式、表格、引用、数值与改写前完全一致

---

## 7. 不要做的事

- 不要新增章节、段落、图表、参考文献
- 不要新增或删除任何数值
- 不要为了"更通顺"而修改公式说明里的变量名
- 不要改写英文段落（除非用户显式要求）
- 不要修改参考文献相关命令和条目
- 不要对一个文件做一次性大批量改写，必须分批 `multi_edit`
- 不要在没有项目专有术语表的情况下擅自统一术语；先与用户确认
